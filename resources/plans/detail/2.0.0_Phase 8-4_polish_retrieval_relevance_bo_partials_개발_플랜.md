### Backlog #1 — retrieve_documents false positive (사용자 가시 잡음)

Phase 8-1 smoke 시나리오 A''' (`2025년 1월 1일부터 100일 후 날짜는?`) 에서 발견:

```
agent step 0: tool='retrieve_documents' args={'query': '2025년 1월 1일 100일 후 날짜'}
              → is_failure=False kind=None  (← false negative — 무관 청크가 의미 매치 통과)
[1] zfFzDCABT8zz-개인정보 처리방침 (2025_V.1)-...pdf  (← 무관 출처가 sources 로 노출)
```

근본 원인: `_has_meaningful_match` 의 longest meaningful tier 판정이 **일반 시간/숫자 토큰** (`2025년`, `100일`, `1월` 등) 을 그대로 의미 토큰으로 인정. `_LOW_SIGNAL_TOKENS` (`비교`, `얼마` 등) 에 없으면 모두 의미 토큰. "2025년" / "100일" 같은 토큰이 임의 PDF (개인정보 / 취업규칙 등) 에 자연 등장 → false positive.

기능적 영향: 사용자가 답변과 무관한 PDF 를 "근거 출처" 로 인지. 답변 자체는 (workflow 가 답한 경우) 정상이라 더 큰 잡음.

    - `rf'^기간{_PARTICLE_SUFFIX}$'` — `기간` / `기간이` 등 (같은 이유).

### 1. Retrieval relevance — regex 기반 일반화 (LLM judge X)

옵션:
- (A) `_LOW_SIGNAL_TOKENS` 단순 enumeration 확장.
- (B) regex 패턴 추가 (숫자+단위) — 본 plan 채택.
- (C) LLM-based relevance judge — chunk 가 query 와 의미적으로 관련 있는지 LLM 에 묻기.
- (D) embedding cosine similarity threshold — chunk 자체 임베딩 vs query 임베딩.

**(B) 채택**:
- A 는 enumeration 한계 — `2025년` / `1년` / `100년` 다 따로 등록 불가능.
- C 는 호출당 LLM 추가 호출 → retrieve 비용 폭증. 8-2 의 비용 분리 의도와 충돌.
- D 는 retrieval layer 의 큰 변경. 본 polish 범위 밖.
- **B**: regex 한 줄 추가로 숫자+단위 패턴 일반화. 비용 0, 의존성 0, 회귀 가능성 낮음.

설계 §6 의 한계 ("의문/일반어가 우연히 매치"), 7-4 smoke 의 잡음 모두 같은 뿌리 — 일반어 카테고리 추가 정책으로 해결.

### 2. `_LOW_SIGNAL_TOKENS` 와 `_LOW_SIGNAL_PATTERNS` 의 분리

- `_LOW_SIGNAL_TOKENS`: 명시 enumeration — 도메인 무관한 의문/지시 표현 (`비교`, `얼마`).
- `_LOW_SIGNAL_PATTERNS`: regex 일반화 — 숫자+단위 (`\d+년`, `\d+일`).

두 개로 나누는 이유:
- 의도가 다름. 명시 enumeration 은 "고유한 단어" 리스트. regex 는 "패턴" 으로 잡는 일반화.
- 운영 중 추가 시 명시 enumeration 추가는 단순한 backlog (단어 한 개 추가), regex 패턴 추가는 영향 범위 검토 필요 (false positive 가능성).
- 두 분리로 향후 운영자가 BO 에서 `_LOW_SIGNAL_TOKENS` 만 추가 가능한 구조 (Phase 9+) 로 확장 여지.

### 3. BO partials — `bo.js` 단일 파일 + `data-*` attribute 규약

옵션:
- (A) 페이지별 인라인 그대로 — 현재 상태.
- (B) 여러 partial template (`_form_shake.html` / `_bulk.html` / `_alert.html`) — 각 페이지가 include.
- (C) 단일 `bo.js` 파일 + base.html 에 한 번 로드 — 본 plan 채택.

**(C) 채택**:
- `bo.js` 가 DOMContentLoaded 시점에 `data-*` attribute 기반으로 자동 동작.
- 페이지 작성자가 별도 호출 / include 불필요 — 마크업 규약만 따르면 됨.
- 기존 `bo.css` 가 이미 공용이라 패턴 일관.
- partial 하나당 include 한 줄씩 늘어나면 페이지마다 boilerplate.

trade-off:
- ✓ 페이지 추가 시 boilerplate 0.
- ✓ defer 로딩이라 페이지 first paint 영향 미미.
- ✗ JS 가 한 파일에 모이는 게 너무 커지면 후속 분리 필요 — 현재 3 패턴 + ~150 줄이라 유지 가능.

### 4. 마크업 규약 — `data-bulk-*` attribute

기존 `qa-bulk-*` / `router-bulk-*` 클래스는 페이지별 다른 이름. `bo.js` 가 일반화하려면 공통 hook 필요.

채택 규약:
- `data-bulk-page="<id>"` — 페이지 컨테이너. `<id>` 는 page-edit-toggle 짝 매칭용 (`router` / `qa-logs` 등).
- `data-bulk-row` — 선택 가능한 행/카드 (page 컨테이너 안).
- `data-bulk-check` — 숨김 checkbox (form 외부 참조).
- `data-bulk-toolbar` — 일괄 액션 툴바 (page 컨테이너 안).
- `data-bulk-action` — 액션 button (form="..." 으로 폼 외부 참조, toolbar 안).
- `data-bulk-confirm="<msg>"` — 위험 액션 confirm 메시지.
- **`data-bulk-edit-toggle data-bulk-target="<id>"` — 수정/완료 토글 button. page 컨테이너 밖 (header) 에 위치 가능 — `data-bulk-target` 의 값이 짝지어진 `data-bulk-page` 의 `<id>` 와 일치해야 매칭됨 (P2-3 보강).**
- `data-bulk-select-all` — 전체 선택/해제 button (toolbar 안).
- `data-bulk-count` — 선택 카운터 (textContent 자동 갱신, toolbar 안).

이 규약을 따르면 page 마크업만 바꿔서 동일 JS 가 동작. edit-toggle 은 header 같은 page 컨테이너 밖 위치 허용.
