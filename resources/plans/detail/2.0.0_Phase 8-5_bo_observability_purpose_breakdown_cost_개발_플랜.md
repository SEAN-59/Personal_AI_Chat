# 2.0.0 Phase 8-5 — BO Observability: Purpose Breakdown & Cost 개발 플랜

## Context

Phase 8-4 머지 직후 (2026-04-29) 시점:

- Phase 8-2 가 `TokenUsage.purpose` 필드를 도입해 7 호출 사이트 별 분류가 DB 에 쌓이고 있지만, **BO 대시보드 (`/bo/`) 가 이 데이터를 활용하지 않음**. 일별 합계 / 전체 합계만 노출하고 purpose 별 분해는 없음.
- 운영자가 "agent 추론 비용이 single_shot 답변보다 얼마나 큰가" / "rewriter 가 전체 토큰의 몇 %인가" 같은 핵심 비용 분석을 BO 에서 못 함.
- `TokenUsage` 에 **cost 필드 없음** — 토큰 수만 있고 USD 환산 안 됨. 운영자가 비용 추적할 때마다 모델 단가표를 외부에서 봐야 함.
- Phase 4-4 의 OpenAI Admin API widget 이 cost 를 보여주지만 **외부 청구 기준** (할인 / 리베이트 / 배치 할인 반영) — `TokenUsage` 의 자체 추정치와 데이터 소스가 다름. 두 표면이 분리돼 있어 비교 불가.

Phase 8-5 의 목표는 **TokenUsage 에 cost_usd 필드 추가 + BO 대시보드의 purpose 별 분해 / cost 노출** 으로 운영자가 BO 한 화면에서 자체 추정 비용 / purpose 분해 / 일별 추이를 모두 볼 수 있게 만드는 것.

> 이 plan 은 8-3 PR (#56) out-of-scope 의 backlog `#2 (BO 대시보드 purpose 분해)` + `#6 (TokenUsage cost_usd 환산)` 을 합쳐 처리. 두 항목 모두 BO observability 영역이라 같은 SQL / UI 레이어를 건드림 — 묶는 게 자연스러움.

## Scope

**포함**

### 1. TokenUsage cost_usd 필드 + 모델 단가 매핑

본 plan 의 cost 노출 범위는 **`TokenUsage` 에 기록된 채팅 LLM 호출 기준** (P2 보강). 즉 Phase 8-2 의 7 호출 사이트 (`single_shot_answer / query_rewriter ×3 / workflow_extractor / workflow_table_lookup / agent_step / agent_final`) 만. **embedding 호출 (`files/services/embedder.py`) 과 reranker 호출 (`chat/services/reranker.py`) 은 본 plan 범위 밖** — 두 사이트가 `record_token_usage` 를 부르지 않아 `TokenUsage` 에 row 자체가 안 남음. 두 사이트의 cost 통합은 record 호출 추가 + purpose 추가 + migration 영향이 따로 있어 별 plan (`Phase 8-7` 후보 — out of scope 명시).

- `chat/services/openai_pricing.py` **신규**:
  - `MODEL_PRICING: Mapping[str, Tuple[Decimal, Decimal]]` — 모델명 → `(input_per_1k, output_per_1k)` 단가 매핑. 2026-04 기준 OpenAI 공식 가격표 — **`record_token_usage` 가 실제 호출하는 모델만 등록** (P2 보강 — embedding 모델은 호출 사이트가 없어 제외):
    - `gpt-4o-mini`: input $0.00015, output $0.00060 per 1K tokens
    - `gpt-4o`: input $0.0025, output $0.0100
    - 그 외 후보는 운영 데이터 보고 추가 (예: gpt-4o-2024-08-06 같은 dated 변형).
    - **`text-embedding-3-small` 미등록** — embedder.py 가 record_token_usage 안 부르므로 단가 추가해도 의미 없음. 8-7 에서 embedder 통합 시 추가.
  - `compute_cost_usd(model: str, prompt_tokens: int, completion_tokens: int) -> Decimal` — 단가 lookup × 토큰 / 1000. 등록되지 않은 모델은 **0 + warning 로그** (fail-silent — 비용 추적이 답변 자체를 막지 않게).
  - module docstring 에 "단가 변경 시 검토 항목" 메모 (가격표 인용 / 단위 / 변경 적용 시점 정책 / **embedding · reranker 비용은 별 plan 의 범위라 미포함**).
- `chat/models.py`:
  - `TokenUsage.cost_usd: DecimalField(max_digits=12, decimal_places=6, default=Decimal('0'))` 추가. Decimal 채택 — float 정밀도 누적 오차 방지.
- `chat/migrations/0013_tokenusage_cost_usd.py` (자동 생성):
  - `AddField` + default `0`. **별도 RunPython backfill 없음** — 과거 row 는 0 으로 둠 (cost 추정이 사후에 정확히 계산되지 않을 가능성 — Phase 4-4 widget 의 외부 cost 와도 별개라 backfill 가치 약함).
- `chat/services/single_shot/postprocess.py`:
  - `record_token_usage(model, usage, *, purpose=...)` 가 `compute_cost_usd` 호출 → cost row 에 자동 저장. 시그니처 변경 없음.

### 2. BO 대시보드 purpose 분해 + cost 노출

## 핵심 설계 결정

### 1. 단가 저장 위치 — **코드 상수 (Python module)**

옵션:
- (A) DB 모델 (`ModelPricing` singleton 또는 row 별).
- (B) 환경변수.
- (C) **Python module 상수 → 채택**.

**(C)** 이유:
- 단가 변경 빈도 낮음 (OpenAI 가격 발표 시 ~분기에 한 번).
- DB 도입은 audit 추적 / migration 부담 + BO 편집 UI 까지 가야 의미 — over-engineering.
- 환경변수는 다중 모델 단가를 표현하기 부적절.
- 코드 변경 + 배포로 충분. 변경 시점이 git history 에 남는 게 audit 으로 충분.

### 2. cost 계산 시점 — **record 시점 DB 저장**

옵션:
- (A) `record_token_usage` 시점에 cost 계산 후 row 에 저장 → **채택**.
- (B) view 단에서 매번 계산 (cost_usd 필드 없음).

**(A)** 이유:
- 정확한 시점 cost 보존 — 단가가 후에 바뀌어도 과거 row 의 cost 는 그 시점 단가 기준 (실제 회계 의미).
- BO 대시보드 SQL 이 단순해짐 — `Sum('cost_usd')` 한 줄.
- Phase 4-4 widget 의 외부 cost 와 비교할 때 "본 시스템 추정" 의 시점이 명확.

trade-off:
- ✗ 단가 보정 필요 시 (예: 실수로 잘못된 단가 등록) 과거 row 가 옛 값. 하지만 보정 시점의 신규 row 부터 정확해지면 충분.

### 3. cost 의 데이터 타입 — **Decimal**

옵션:
- (A) `FloatField` — 단순.
- (B) **`DecimalField(max_digits=12, decimal_places=6)` → 채택**.

**(B)** 이유:
- 토큰당 cost 가 매우 작음 (gpt-4o-mini input: $0.00015 / 1K = $0.00000015 / token). float 누적 오차가 운영 N개월 단위로 쌓이면 무시 못함.
- 회계 / 청구 비교 시 정확도 우선.
- max_digits=12, decimal_places=6 → 최대 $999,999.999999 까지 표현.

### 4. 미등록 모델 처리 — **fail-silent (0 + warning)**

옵션:
- (A) `KeyError` raise — 새 모델 도입 시 즉시 발견.
- (B) **0 + warning 로그 → 채택**.

**(B)** 이유:
- 비용 추적이 답변 자체를 막지 않게. record_token_usage 가 try/except 로 감싸 있긴 하지만, 답변 path 에서 fail-loud 는 부담.
- warning 로그가 미등록 모델 시그널 — 운영자가 발견 시 코드 추가.
- BO 대시보드의 cost 합계가 약간 underestimate 될 수 있지만, "0 + warning 로그" 의 시그널이 더 안전.

### 5. Phase 4-4 widget 과의 관계 — **의도적 분리**

본 plan 의 cost 는 **`TokenUsage` 에 기록된 채팅 LLM 호출 비용의 추정치** (P3 보강 — 전체 OpenAI 호출 비용이 아님):
- Phase 8-2 의 7 호출 사이트 (single_shot_answer / query_rewriter ×3 / workflow_extractor / workflow_table_lookup / agent_step / agent_final) 에 한정.
- 모델 단가 매핑 × 토큰 수 (단순 곱).
- 할인 / 리베이트 / 배치 할인 / 캐시 hit 환급 미반영.
- **embedding (`files/services/embedder.py`) / reranker (`chat/services/reranker.py`) 호출 비용은 미포함** — 두 사이트가 record_token_usage 를 부르지 않아 TokenUsage 에 row 없음 (8-7 후보).

Phase 4-4 widget 의 cost 는 **외부 청구 기준** (OpenAI Admin API):
- 실제 OpenAI 가 청구한 금액 (모든 API 호출 — 채팅 / embedding / 그 외).
- 할인 등 모두 반영.
- "OpenAI 가 청구하는 비용" 의 정확 값.

두 데이터 소스 분리 의도 — 운영자가 두 값을 비교해 (a) 자체 추정과 실제 청구 차이 (할인 효과), (b) 누락된 호출 사이트 (BO cost 가 외부 청구의 일부만 커버하는 정상 차이) 를 인지 가능. 본 plan 은 통합하지 않고 같은 BO 페이지에서 분리 표시 (대시보드 본 영역 vs 모달). README / dev log 도 같은 표현 — "TokenUsage 기록 호출 기준 추정" 으로 일관 사용.

### 6. purpose 한국어 라벨 mapping — **view 안의 dict**

`single_shot_answer` 같은 영문 코드 상수를 한국어 ("single-shot 답변") 로 바꿔 표시. 위치 옵션:
- (A) **view 안의 dict → 채택**.
- (B) `chat/services/token_purpose.py` 의 `LABEL_KO` 같은 새 dict.

**(A)** 이유:
- 운영자-facing 표시는 BO 영역. 도메인 로직 (`token_purpose.py`) 과 분리.
- 한국어 라벨 변경 시 view 한 곳만 수정.
- 향후 다국어 / i18n 시 Django translation 으로 자연스럽게 이동.

## 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| 단가 매핑 누락된 모델 → 신규 row cost 0 | `compute_cost_usd` 가 warning 로그. BO 대시보드의 `unknown` purpose / 알 수 없는 모델 row 에서 cost 0 이 비정상 큰 비율이면 시그널. 운영자가 발견 시 `MODEL_PRICING` 에 추가. |
| 단가가 OpenAI 가격 변경에 안 맞음 | 단가 변경 빈도 낮음. 변경 시 `openai_pricing.py` 수정 + 코멘트에 변경 시점 기록. 과거 row 는 옛 단가 기준 그대로 — 정확한 시점 cost 보존. |
| Decimal 연산 비용 | Python Decimal 은 float 보다 느리지만 record_token_usage 호출당 1회 (~분당 1~2 회) 라 무시 가능. |
| 마이그레이션 시점 production DB 잠금 | `cost_usd` 추가는 default=0 + nullable 아니라 ALTER TABLE 짧음. PostgreSQL 에서 instant 또는 mini-lock. 운영 윈도우 < 5초 예상. |
| BO 대시보드 SQL 추가 query 부하 | `purpose_rows` 가 같은 테이블 GROUP BY purpose. db_index=True (Phase 8-2) 라 비용 미미. 기존 `daily_rows` 의 GROUP BY date 와 비슷한 비용. |
| Phase 4-4 widget 의 cost 와 다른 값 노출로 운영자 혼란 | 두 영역에 데이터 소스 라벨 명시 — 본 plan 이 추가하는 부분에 "채팅 LLM 호출 기준 자체 추정 (embedding · reranker 미포함)" 을 명시 (대시보드 카드 / 표 caption). 4-4 모달은 "OpenAI 청구 기준" 명시. dev log 에 분리 의도 + 8-7 이 embedder / reranker 통합 작업임을 명시. |
| BO cost 가 OpenAI 실제 청구의 일부만 반영 (embedding / reranker 누락) | 본 plan 은 채팅 LLM 호출만 cost 추정. embedder.py / reranker.py 가 record_token_usage 안 부르는 현 구조 그대로 — 통합은 8-7. 운영자가 "BO cost < Phase 4-4 외부 청구" 로 보이는 게 정상이라는 점을 캡션에 명시. |
| 한국어 라벨 dict 가 새 purpose 추가 시 누락 | 미등록 purpose 는 dict.get(purpose, purpose) 로 fallback (영문 코드 그대로 표시). 후속 Phase 가 새 purpose 추가 시 라벨도 같이 추가. |
