# 0.4.0 Phase 8-6 — Agent Settings Expansion: Extra Limits & Audit Log 개발 플랜

## Context

Phase 8-5 머지 직후 (2026-04-29) 시점:

- Phase 8-3 가 `AgentSettings` singleton 을 도입했지만 BO 노출 한도는 **`max_iterations` / `max_low_relevance_retrieves` 두 개만**. `react.py:55-56` 의 `MAX_CONSECUTIVE_FAILURES = 3` / `MAX_REPEATED_CALL = 3` 은 여전히 코드 상수 — 운영 incident 시 즉시 조정 불가.
- `AgentSettings` 변경 이력이 **전혀 추적되지 않음**. `updated_at` 만 있어 마지막 변경 시각만 알 뿐, "누가 / 무엇을 / 언제 / 어떤 값으로" 변경했는지는 git/log 에도 없음. 운영 incident 후 원인 분석 불가.

Phase 8-6 의 목표는 (1) **`MAX_CONSECUTIVE_FAILURES` / `MAX_REPEATED_CALL` 의 BO 노출** + (2) **`AgentSettings` 변경 audit log** 도입. 8-3 의 control plane 을 완성하고 운영 추적성을 확보.

> 이 plan 은 8-3 PR (#56) out-of-scope 의 backlog `#4 (MAX_CONSECUTIVE_FAILURES / MAX_REPEATED_CALL BO 노출)` + `#5 (AgentSettings 변경 audit log)` 를 합쳐 처리. 두 항목 모두 `AgentSettings` 모델 영역이라 같은 layer 를 건드림 — 묶는 게 자연스러움.

### 2. `AgentSettings` 변경 audit log (#5)

- `chat/models.py`:
  - `AgentSettingsAudit` 모델 신규 — `AgentSettings` 변경 이력 추적용:
    - `changed_at: DateTimeField(auto_now_add=True, db_index=True)`

    - `changes: JSONField(default=dict)` — `{field: {'old': X, 'new': Y}, ...}` (Decimal 은 str 으로).
    - `snapshot: JSONField(default=dict)` — 변경 후 전체 상태 (`enabled / max_iterations / max_low_relevance_retrieves / max_consecutive_failures / max_repeated_call`).

- `chat/migrations/0015_agentsettingsaudit.py` (자동 생성) — `CreateModel`. 별도 RunPython 없음 (audit 은 신규 row 만 누적).
- `bo/views/agent.py`:
  - `agent_view` 의 POST 분기에서 **form 바인딩 전에** DB 의 현재 값을 별도 dict 로 캡처 — `old_values = {f: getattr(settings_obj, f) for f in AUDIT_FIELDS}` (P2-1 보강 — Django `ModelForm.is_valid()` 가 `cleaned_data` 를 `form.instance` 에 반영하므로 `form.is_valid()` 후 `form.instance` 에서 old 를 읽으면 `old = new` 가 되어 변경 감지 실패).
  - `form = AgentSettingsForm(request.POST, instance=settings_obj)` 는 그 다음 단계.
  - `form.is_valid()` + `saved = form.save()` 후 `saved` 의 새 값과 `old_values` 비교 → 변경된 필드만 `changes` dict 에 담음.

  - `request.user.is_authenticated` 이면 `changed_by=request.user`, 익명이면 NULL.
  - 전체 `transaction.atomic()` 으로 form.save() + audit 생성을 한 트랜잭션 내 묶음 — 둘 중 하나 실패 시 롤백.
  - GET 분기에서 최근 N개 (10) audit row 조회 → context `recent_audits` 로 전달.
  - **module-level 상수 `AUDIT_FIELDS = ('enabled', 'max_iterations', 'max_low_relevance_retrieves', 'max_consecutive_failures', 'max_repeated_call')`** — 새 필드 추가 시 한 곳만 갱신.

## 핵심 설계 결정

### 1. 두 한도 추가 — Phase 8-3 패턴 그대로

`max_iterations` / `max_low_relevance_retrieves` 와 동일한 layer 에 두 필드 추가. DB CheckConstraint + Form validator + runtime sanity check 삼중 가드. 시간 절약 — 새 패턴 도입 없이 기존 인프라 재사용.

### 2. Audit 저장 위치 — **view layer** (signal 아님)

옵션:
- (A) Django `pre_save` / `post_save` signal — 모델 layer 에서 자동 캐치.
- (B) **`agent_view` POST 분기 → 채택**.

**(B)** 이유:
- `request.user` 자연 접근 — signal 은 user 흐름 추적이 어려움 (thread-local 같은 hack 필요).
- 변경 컨텍스트 명확 — POST 가 BO UI 의 명시 액션, signal 은 모든 save 캐치 (data migration / shell save 등 잡음).
- 본 plan 은 BO UI 변경만 audit 대상 — 다른 경로는 의도적 미추적 (8-3 의 `get_solo()` 같은 자동 생성도 audit 안 남게).

trade-off:
- ✗ shell / Django admin 직접 변경은 audit 안 됨 — 정책상 의도적.
- ✓ 운영자 BO 액션만 깔끔히 추적.

### 3. Audit 모델 형태 — **changes + snapshot 둘 다**

옵션:
- (A) snapshot 만 — read 시 이전 row 와 비교해서 changes 도출.
- (B) changes 만 — 가벼움, 다만 시점 전체 상태 모름.
- (C) **둘 다 → 채택**.

**(C)** 이유:
- changes = "이번에 뭐가 바뀌었나" — UI 표시용 (가독성 ↑).
- snapshot = "당시 전체 설정 상태" — incident 분석 시 "그 시점에 enabled 가 무엇이었나" 같은 질문에 즉답.
- 두 dict 의 저장 비용 미미 (각 ~100 bytes).
- read query 단순 — 이전 row JOIN 불필요.

### 4. 변경된 필드만 `changes` 에 — empty save audit 노이즈 차단

**`old_values` 는 form 바인딩 / `is_valid()` 호출 전에 DB 의 현재 값으로 별도 dict 캡처** (P2-1 정합 — Django `ModelForm.is_valid()` 가 cleaned_data 를 form.instance 에 반영하므로 form 으로부터 old 를 읽으면 `old = new` 가 되어 변경 감지 실패. `Scope` §2 의 view 흐름과 동일 계약). save 후 `saved` 객체의 새 값과 `old_values` 비교해 **실제 변경된 필드만** changes dict 에 포함. 변경된 필드 0개면 `AgentSettingsAudit` row 자체를 만들지 않음. 운영자가 단순히 폼만 열고 "저장" 누른 케이스가 audit 에 잡음으로 남지 않게.

### 5. `changed_by` 의 `on_delete=SET_NULL`

운영자 계정 삭제 시 audit row 삭제 안 함 (CASCADE) — audit 은 영구 보존이 원칙. NULL 로 두고 UI 에서 "(삭제된 사용자)" 표시.

### 6. BO 노출 — 같은 페이지 하단 (별 페이지 아님)

옵션:
- (A) `/bo/agent/` 페이지 하단에 audit 표 — **본 plan 채택**.
- (B) `/bo/agent/audit/` 별 페이지 + 페이지네이션 / 필터.

**(A)** 이유:
- 운영자가 설정 변경 후 즉시 "내가 방금 무엇을 바꿨나" 확인 가능 — 같은 페이지가 자연스러움.
- 본 plan 은 최근 10건 만 — 충분.
- full history 는 별 plan (out-of-scope) — 운영 데이터 더 쌓이고 필요해지면.

### 7. `MAX_REPEATED_CALL` 의 의미 — 정책 강화 X, BO 노출만 + min=2 가드

8-3 의 `repeated_call` 차단 정책은 "동일 (tool, args) 호출은 즉시 차단" — 1회만 허용. **반복 시도는 `tool_calls` 에 record 안 되고 failure observation 만 추가됨** (`react.py` 의 `record_tool_call` skip 분기). 따라서 `state.repeated_call_count(last.name, last.arguments)` 는 첫 정상 호출의 `1` 그대로 머물고 절대 `2` 이상으로 증가 안 함.

결과: `_decide_termination` 의 `repeated_call_count >= MAX_REPEATED_CALL` 분기 (Phase 7-4 backward compat) 는 **모든 max_repeated_call 값 (≥2)** 에서 사실상 도달 불가. BO 에서 운영자가 값을 2 / 3 / 10 어떻게 변경해도 실제 종료 동작 변화 없음. 본 plan 은 BO 노출만 추가하고 정책 자체는 8-3 그대로 — 후속 plan 에서 차단/카운터 정책 통합 시 의미 부여 예정.

**`max_repeated_call=1` 만 의미가 다름** (P2-3 보강): `repeated_call_count` 가 첫 정상 호출도 `1` 로 세므로 max=1 이면 차단 정책 우회 케이스가 아닌데도 첫 tool call 직후 즉시 NO_MORE_USEFUL_TOOLS 발동. 정책 의도 ("repeat 시 종료") 와 충돌.

해결: **DB CheckConstraint + Form validator 의 min 을 2 로 강제**. 운영자가 1 입력 못 함. default=3 그대로 유지. min=2 부터는 카운터 의미상 어차피 도달 불가라 동작 동일 — `max_repeated_call=1` 거부만 본 가드의 역할.

대안 검토:
- (A) `repeated_call_count` 의미 재정의 ("첫 호출 이후 repeat 수만 카운트") — 8-3 의 차단 분기에 영향 ↑, 위험.
- (B) `_decide_termination` 의 `>=` 를 `>` 로 변경 — 코드 한 줄. 다만 `>=N` 패턴이 다른 분기 (`max_iterations`) 와 일관 깨짐.
- **(C) min=2 validator → 채택** — 의미는 그대로, 가드 layer 로 처리.

## 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| `max_repeated_call` 가 `_decide_termination` 의 분기에서 사실상 도달 불가 (8-3 의 동일 호출 차단으로 1회만 허용) | 정책 변경 의도 X — BO 노출만. 운영자가 값 조정해도 실제 동작 변화 거의 없음. 본 plan 의 audit 가 변경 시도 시각 기록. 후속에서 정책 통합 검토 시 audit 가 base. |
| audit 모델 row 가 운영 N개월 후 너무 많이 쌓임 | 본 plan 은 retention 미도입 — 일단 영구 저장. BO 노출은 최근 10건 만이라 row 수와 무관. 1년 후 N=수천이면 충분히 작은 수 (운영자 변경 빈도 낮음). retention 도입은 후속 plan. |
| `request.user` 가 `AnonymousUser` 인 경우 `changed_by=NULL` 로 audit 의 user 추적 의미 약화 | BO 페이지에 인증 강제는 본 plan 범위 밖 (별 보안 작업). audit 표에 "(익명)" 표시. 운영 환경에서 BO 인증을 의무화하면 자연 해소. |
| Phase 8-3 의 SQL constraint name 이 `agentsettings_max_iterations_range` / `agentsettings_max_low_relevance_range` 인데 본 plan 의 새 constraint name 충돌 가능 | constraint name 이 `agentsettings_max_consecutive_failures_range` / `agentsettings_max_repeated_call_range` 로 distinct. makemigrations 가 auto-generate 한 이름 검토 후 명시 rename 가능. |
| `AgentSettingsAudit.changes` 가 JSON 이라 enum 값 / Decimal 직렬화 주의 | 본 plan 의 변경 대상 5 필드는 모두 bool / int — JSON 호환. 향후 새 필드 (Decimal / Date 등) 추가 시 직렬화 정책 검토 필요. |
| audit 생성이 form.save() 와 별 transaction → race 가능 | `transaction.atomic()` 으로 둘 묶음. POST 분기 view 안에서 한 트랜잭션. |
