# 2.0.0 Phase 5 — Workflow Core 개발 플랜

## Context

Phase 3 에서 `chat/workflows/{core,domains}/__init__.py` 뼈대만 만들고 비워뒀다. Phase 4 (라우팅 Core → BO 관리 → retrieval 맥락화 → 사용량 위젯) 를 마친 지금, 실제 workflow 본체를 올리기 전에 **도메인 규칙과 무관한 범용 계산·정규화·검증·포맷팅 레이어** 를 먼저 깐다.

이 단계의 목적은 Phase 6 에서 "퇴직금 계산 / 연차 산정 / 급여 평균" 같은 도메인 workflow 를 붙일 때, **공통 엔진이 이미 준비돼 있도록** 하는 것. 범용 유틸 없이 도메인 workflow 부터 쓰기 시작하면 곧바로 "계산 함수에 회사 규정이 섞이는" 오염 패턴에 빠지기 때문이다.

참고: Phase 4-1 결정대로 graph 의 `workflow` route 는 **여전히 single_shot 으로 포워딩**한다. 이번 Phase 는 **service layer 라이브러리 추가만** 하고 graph / view / BO / DB 는 건드리지 않는다.

설계 문서: `resources/plans/2.0.0_Phase 5_Workflow_Core_개발_설계.md`.

## 핵심 설계 결정

### 1. 모듈 의존 방향 (한 방향만 허용)

```
result.py
   ↑
validation.py
dates.py         (서로 독립)
numbers.py       (서로 독립)
formatting.py    (서로 독립)
   ↑
base.py          (네 개 모두 참조 가능)
```

- `result.py` 는 **타입만** 정의, 다른 core 모듈 import 금지.
- `validation.py` 는 `result.ValidationResult` 만 import.
- `dates.py` / `numbers.py` / `formatting.py` 는 **서로 import 하지 않는다.** 필요하면 호출부에서 조합.
- `base.py` 가 네 모듈의 진입점을 조립한다.

이유: 순환 import 차단 + Phase 6 에서 도메인 모듈이 어느 한 core 모듈만 얇게 의존하도록 유도.

### 2. `ValidationResult` 와 `WorkflowResult` 의 관계

두 타입 모두 `frozen dataclass`. 불변 → 조합·합성 안전.

```python
@dataclass(frozen=True)
class ValidationResult:
    ok: bool
    missing_fields: tuple[str, ...] = ()
    errors: tuple[str, ...] = ()

    @classmethod
    def success(cls) -> 'ValidationResult': ...
    @classmethod
    def fail(cls, *, missing=(), errors=()) -> 'ValidationResult': ...


class WorkflowStatus(str, Enum):
    OK = 'ok'
    MISSING_INPUT = 'missing_input'
    INVALID_INPUT = 'invalid_input'
    UNSUPPORTED = 'unsupported'


@dataclass(frozen=True)
class WorkflowResult:
    status: WorkflowStatus
    value: Any | None = None
    details: Mapping[str, Any] = field(default_factory=dict)
    missing_fields: tuple[str, ...] = ()
    warnings: tuple[str, ...] = ()
```

- `ValidationResult` 는 **입력 단계 결과** — workflow 내부에서 쓰이고 사용자까지 가진 않는다.
- `WorkflowResult` 는 **workflow 최종 반환형** — 응답 조립 레이어가 이걸 사용자 문자열로 변환.
- `status` 는 문자열 enum 이라 JSON 직렬화·로그·DB 저장이 전부 안전.

### 3. 검증 스타일: "검증 결과 객체" 우선

설계 §5-4 기준. 단 `raise` 를 아예 금지하진 않음 — **경계선을 문서화**:

| 상황 | 처리 방식 | 예 |
|---|---|---|
| 사용자 입력 부족·형식 오류 | `ValidationResult.fail(...)` | 입사일 누락 |
| 도메인 규칙 어긋남 (예: 시작일 > 종료일) | `ValidationResult.fail(errors=['...'])` | 계약 날짜 역순 |
| 프로그래밍 오류 (타입 불일치, 구조 손상) | `raise TypeError / ValueError` | `days_between(None, ...)` |
| 외부 의존 실패 (core 에는 없음) | 상위 레이어 예외 | — |

core 내 helper 는 **모두 검증 결과 객체를 돌려주는 쪽으로** 통일한다. `dates.ensure_date_order` 가 대표 예시.

### 4. 날짜 파싱 정책

받을 포맷 (한국어 사용자가 흔히 쓰는 형태):

- `2025-01-31`, `2025/01/31`, `2025.01.31`
- `25-01-31`, `25.01.31`  (2자리 연도 → 2000+ 으로 처리)
- `2025년 1월 31일`, `2025년 01월 31일`
- 이미 `date` 또는 `datetime` 인스턴스 → 그대로 반환

**미지원** (Phase 5 out-of-scope):
- 상대 표현 (`오늘`, `어제`, `지난주`)
- 한자 연월 표기
- 타임존 포함 문자열 (필요하면 상위 레이어에서 TZ 를 떼고 넘김)

파싱 실패 시 `ValueError` 로 raise — 호출부가 `try/except` 로 감싸 `ValidationResult.fail` 을 만들도록. (파싱 함수가 직접 `ValidationResult` 를 돌려주면 "날짜인지 아닌지 모르는 상황에서 계산도 해야 하는" 함수 복잡도가 커짐.)

### 5. 숫자 파싱 정책

`parse_int_like("1,234,567원")` 은 1234567 반환. 규칙:

- 콤마 제거
- 공백 제거
- 한국어 단위 접미어 `원` / `개월` / `년` / `일` 하나까지만 제거 (정규식 `\s*(원|개월|년|일)$`)
- 남은 문자열이 순수 정수로 파싱 가능해야 함 (부호 허용: `-`, `+`)

**미지원**:
- `3.14` 같은 실수 (돈은 정수 원 단위 전제)
- `천만`, `3억` 같은 한글 수사 — Phase 6 이후 필요해지면 확장
- 통화 접두어 (`$`, `₩`)

`parse_money` 는 `parse_int_like` 의 얇은 alias — 현재는 동일 동작이지만 "돈 의미" 의도를 드러내기 위해 분리. Phase 6 에서 0 이하 금액 거부 같은 규칙이 붙을 때 여기에 쌓는다.

### 6. formatting 정책

- `format_currency(1234567)` → `"1,234,567원"`
- `format_date(date(2025,1,31))` → `"2025-01-31"`
- `format_duration(days=30)` → `"30일"`, `format_duration(months=3)` → `"3개월"`, `format_duration(years=1)` → `"1년"`
- `format_duration(years=1, months=2, days=3)` → `"1년 2개월 3일"`

**금지**: 자연어 답변(예: "평균 급여는 1,234,567원 입니다") 조립. 그건 response layer 책임.

### 7. BaseWorkflow — Protocol 사용

설계 §9 "abstract class 까지 강제할 필요 없다" 에 맞춰 `typing.Protocol` 로만 정의.

```python
class BaseWorkflow(Protocol):
    """4단계 계약: prepare → validate → execute → result.

    실제 도메인 workflow 는 이 Protocol 을 따르는 클래스/함수면 되고,
    상속을 강제하지 않는다. runtime checkable 로 두어 테스트에서 isinstance
    검사가 가능.
    """
    def prepare(self, raw: Mapping[str, Any]) -> Mapping[str, Any]: ...
    def validate(self, normalized: Mapping[str, Any]) -> ValidationResult: ...
    def execute(self, normalized: Mapping[str, Any]) -> WorkflowResult: ...
```

Phase 6 에서 도메인 workflow 가 이 Protocol 을 충족하기만 하면 된다. 상속은 강제 안 함 → 함수 조합형 workflow 도 허용.

`base.py` 에 정의하는 것:
- `BaseWorkflow` Protocol
- `run_workflow(workflow, raw)` — 네 단계를 엮는 레퍼런스 러너. 검증 실패면 `WorkflowResult(status=MISSING_INPUT or INVALID_INPUT)` 로 변환해 반환.

### 8. 테스트 전략

`chat/tests/test_workflows_<module>.py` 6개 파일. 순수 함수라 빠르게 돌고 DB 불필요.

커버리지 가이드:
- 각 공개 함수당 "정상 1 + 경계 1 + 실패 1" 최소 3 케이스.
- `ValidationResult.combine` 같은 합성 helper 는 "둘 다 OK", "둘 다 실패", "mixed" 3 케이스.
- `parse_date` 는 지원 포맷마다 1 케이스 + 실패 포맷 1 케이스.
- `BaseWorkflow` 는 가상의 "합계 workflow" 하나 만들어 run_workflow 가 4단계를 올바르게 엮는지 확인.

총 60~80 케이스 예상. 모두 수 ms 이내.
