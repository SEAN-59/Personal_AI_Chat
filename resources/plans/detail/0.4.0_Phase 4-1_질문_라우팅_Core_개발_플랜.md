# 0.4.0 Phase 4-1 — 질문 라우팅 Core 개발 플랜

## Context

Phase 3 에서 `chat/services/single_shot/` 으로 파이프라인이 정리됐지만, `chat/graph/nodes/router.py` 는 여전히 `{'route': 'single_shot'}` 만 반환하는 placeholder 다. 모든 질문이 single-shot 으로 흐르고 있어, Phase 5~6 의 workflow (퇴직금·연차 계산 등) 와 Phase 7 의 agent (비교·추천) 가 붙을 자리만 있고 들어갈 길이 없다.

Phase 4-1 의 목표는 **규칙 기반 router 함수를 구현해 `single_shot / workflow / agent` 3분류를 실제로 결정하게 하는 것**. 단, workflow/agent 노드 본체는 Phase 5~7 소관이므로 **이 단계에서 분류된 요청은 모두 single_shot 노드로 내부 포워딩**한다. 사용자 체감은 기존과 동일하고, `state.route` 와 서버 로그에만 의도된 경로가 남는다. Phase 5 가 오픈하는 순간 conditional edge 매핑만 바꾸면 workflow 경로가 살아난다.

Phase 4-2 (BO 기반 rule 관리)는 본 플랜 범위 밖 — 다음 별도 PR 에서 다룬다.

핵심 제약:
- 사용자 응답(JSON / 화면) **회귀 0**
- helper·검색·reranker·prompt·LLM 등 single_shot 내부 **건드리지 않음**
- `chat/services/single_shot/__init__.py` 의 공통 규칙(history / error / token / ChatLog / sources / classification) 유지
- Phase 5 PR 이 최소한만 건드리면 workflow 경로를 open 할 수 있게 구조 설계

## 핵심 설계 결정

### 1. Fallback 전략: 내부 포워딩 (사용자 체감 동일)
- `router_node` 가 `state.route` 에 의도된 경로(`'workflow'` 등)를 정직하게 기록
- conditional edge 매핑:
  ```python
  {
      ROUTE_SINGLE_SHOT: 'single_shot',
      ROUTE_WORKFLOW:    'single_shot',   # Phase 5~6 에서 workflow 노드로 교체
      ROUTE_AGENT:       'single_shot',   # Phase 7 에서 agent 노드로 교체
  }
  ```
- 의도된 경로가 single_shot 이 아닐 때 `router_node` 에서 INFO 로그 한 줄 ("classified as workflow → fallback to single_shot (Phase 5 대기)")
- Phase 5 PR 이 하는 일은 edge 매핑의 `ROUTE_WORKFLOW` 키만 실제 노드로 바꾸고 fallback 로그 문구를 제거하는 것.

### 2. 규칙 기반 분류 함수: keyword substring 매칭
설계 §4-1-7 대로 간단한 `contains` 방식만 Phase 4-1 에 둔다. regex / NLP 없이 `if any(kw in question for kw in WORKFLOW_KEYWORDS)` 수준.

우선순위 (설계 §4-1-3):
1. workflow 키워드 매치 → `workflow`
2. agent 키워드 매치 → `agent`
3. 그 외 → `single_shot`

workflow 가 먼저인 이유: 안정성·비용이 낮음. agent 는 마지막 수단.

**주의 (설계 §4-1-4 / §4-1-5)**: 단순 키워드 오분류 가능성 있음 (예: "복지포인트는 얼마야?" 는 single_shot 이 적절한데 "얼마" 때문에 workflow 로 갈 수 있음). Phase 4-1 은 **명시적으로 이 단점을 감수**하고, Phase 4-2 의 BO rule 과 priority 로 운영에서 조정할 예정.

### 3. Keyword 상수 위치: `question_router.py` 모듈 상수
설계의 키워드 목록을 그대로 가져온다. 모듈 최상단에 두 개의 튜플로:

```python
WORKFLOW_KEYWORDS = (
    '계산', '산정', '얼마', '몇 일', '몇 년', '평균', '합계', '차감',
    '근속', '퇴직금', '연차 계산', '잔여 연차', '급여', '수당',
    '입사일', '퇴사일', '최근 3개월',
)

AGENT_KEYWORDS = (
    '비교', '추천', '유리', '불리', '종합', '예외', '케이스',
    '해석', '충돌', '만약', '내 상황', '어떤 게 나아',
)
```

Phase 4-2 에서 DB 기반 rule 이 들어오면 "DB rule → 못 찾으면 이 상수 fallback" 구조가 됨 (설계 §5-8). 즉 이 상수가 나중에 `fallback layer` 역할을 맡는다.

### 4. `RouteDecision` 최소 필드
설계 §4-1-7 에서 `route`, `reason`, `matched_rules`, `confidence` 중 Phase 4-1 은 **앞의 세 개만** 채운다.

```python
@dataclass(frozen=True)
class RouteDecision:
    route: str              # ROUTE_* 중 하나
    reason: str             # 'workflow_keyword' / 'agent_keyword' / 'default'
    matched_rules: list[str] = field(default_factory=list)   # 매치된 키워드들
```

`confidence` 는 규칙 기반에서 의미 약함 → Phase 4-2 이후 추가.

### 5. state 필드 확장: flat vs nested
flat 으로 간다. 이유:
- `state.route_reason` / `state.matched_rules` 를 그대로 꺼내 로깅·관측에 쓸 수 있음
- TypedDict 에 중첩 객체 넣으면 LangGraph state merge 동작이 꼬일 가능성
- `single_shot/__init__.py` 에 명시한 "state 는 단순 dict" 철학과 일관

```python
class GraphState(TypedDict, total=False):
    question: str
    history: list[dict]
    route: str
    route_reason: str           # 신규
    matched_rules: list[str]    # 신규
    result: Optional[QueryResult]
    error: Optional[str]
```

### 6. route 상수 위치: `chat/graph/routes.py`
graph 가 route 값의 소비처 (conditional edges) 이자 최종 소유자. question_router 는 그 값을 '만들어' 내므로 graph 의 routes 를 import 하는 구조.

```python
# chat/graph/routes.py
ROUTE_SINGLE_SHOT = 'single_shot'
ROUTE_WORKFLOW = 'workflow'
ROUTE_AGENT = 'agent'
ALL_ROUTES = (ROUTE_SINGLE_SHOT, ROUTE_WORKFLOW, ROUTE_AGENT)
```

문자열 리터럴 오타 방지 + Phase 4-2 에서 BO 선택지를 만들 때 이 튜플을 그대로 choices 로 사용.

### 7. 검증 방법: 대표 질문셋 smoke
설계 §6 의 예시 표 + 몇 개 추가 (잡담·no-sources 경계)로 shell 에서 한 번에 돌리는 스크립트. 결과를 Phase 4-1 dev log 에 실제로 출력된 route / reason 과 함께 박아 둔다. 회귀 기준.

```

    print(f'{q:40s} → {d.route:12s} ({d.reason})  {d.matched_rules}')

```

  from chat.graph.routes import ROUTE_SINGLE_SHOT
  from chat.graph.state import GraphState
  from chat.services.question_router import route_question

  logger = logging.getLogger(__name__)

  def router_node(state: GraphState) -> dict:
      decision = route_question(state['question'])
      if decision.route != ROUTE_SINGLE_SHOT:
          logger.info(
              '라우팅: %s (reason=%s, rules=%s) — Phase 5~7 대기 중이라 single_shot 로 포워딩',

          'matched_rules': decision.matched_rules,

## 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| 키워드 오분류 (예: "복지포인트 얼마" → workflow) | 설계 §4-1-4 에서 이미 인지. Phase 4-2 에서 BO rule priority / negative pattern 으로 해결. Phase 4-1 에선 dev log 의 smoke 표에 오분류 사례 기록해 두고 넘어간다 |
| router_node 가 예외 발생 시 graph 가 정지 | `route_question` 은 순수 함수 + 문자열 연산만 — 예외 여지 거의 없음. 만일 생기면 그대로 올려 Django 500 경로로 (Phase 3 의 `chat/services/single_shot/__init__.py` 원칙과 동일) |
