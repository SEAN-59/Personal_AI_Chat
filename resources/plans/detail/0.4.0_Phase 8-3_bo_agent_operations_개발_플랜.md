# 0.4.0 Phase 8-3 — BO Agent 운영 제어 개발 플랜

## Context

Phase 8-2 머지 직후 (2026-04-28) 시점:

- agent runtime 의 동작 한도 (`DEFAULT_MAX_ITERATIONS=6`, `MAX_LOW_RELEVANCE_RETRIEVES=3` 등) 가 모두 `chat/services/agent/react.py:48-56` 의 **코드 상수**. 운영자가 BO 에서 조정 불가 — 변경하려면 코드 수정 + 재배포.
- `agent` route 는 항상 켜져 있음. **비상시 끌 방법 없음** (RouterRule 일괄 disable 외엔 코드 수정밖에 없음).
- agent 가 어떤 도구를 갖고 있는지 운영자가 BO 에서 확인 불가. `tools.all_entries()` 결과는 코드 import 시점에 register 부작용으로만 노출됨.
- Phase 8-2 가 token_purpose 별 분리 집계 데이터를 만들어줬지만 BO 대시보드 / 별 페이지에서 본격 활용은 없음 — "agent 호출 비용이 얼마인지", "최근 agent 가 얼마나 자주 실패했는지" 가 BO 에서 보이지 않음.

이 상태에서 운영 incident 가 발생하면 (예: agent 가 무한 루프로 돌면서 비용 폭증) 즉시 대응이 어렵다. 코드 수정 + 재배포 전까지는 RouterRule 로 agent 라우팅 차단하는 우회뿐.

Phase 8-3 의 목표는 **운영자가 BO 에서 agent 의 동작 범위를 안전하게 조정 / 가시화** 할 수 있는 첫 표면 (1차 control plane) 을 만드는 것.

> 이 plan 은 Phase 8 설계 §7 (BO Agent 운영 제어) 에 직접 매핑. §7-2 의 6 가지 후보 작업 중 첫 4 개 (Agent 설정 페이지 / enable 토글 / max_iterations / tool catalog 읽기 전용) 가 본 plan 의 §Scope 핵심. 나머지 (RouterRule 관리 보강 / 실패 원인 요약) 는 §Out of Scope.

## 핵심 설계 결정

설계 §7-3 의 4 가지 질문 + §7-4 권장 방향에 대한 본 plan 의 답:

### 1. 설정 저장 위치 — **DB (`AgentSettings` singleton)**

설계 §7-3 첫 질문에 대한 답.

옵션:
- (A) DB 모델 (singleton).
- (B) 환경변수 (`.env`).
- (C) Django settings.py 의 모듈 상수.

본 plan 은 **(A)**. 이유:
- 운영자가 BO 에서 즉시 변경 가능 — 본 plan 의 **본 목표**.
- 변경 즉시 반영 (재배포 / 재시작 불필요) — `load_runtime_settings()` 가 매 호출마다 DB SELECT 1회.
- (B) / (C) 는 운영 incident 시 코드 변경 + 배포 절차가 필요해 본 plan 의 incident 대응 가치 무력화.

**Hard singleton 강제** (P2-1 보강): 자체 manager 의 `get_solo()` 가 pk=1 row 가 없으면 생성 후 반환 + **`AgentSettings.save()` override 가 `self.pk = 1` 강제**. PK 유일성만으로는 `objects.create(pk=2, ...)` 가 shell/admin 우회 가능 — save override 가 진짜 차단. `solo` 라이브러리 안 도입 (의존성 비용 vs 효용 — save override 4 줄로 충분).

### 2. 잘못된 설정 방어 — **CheckConstraint + Form validators + runtime sanity 삼중**

설계 §7-3 두 번째 질문에 대한 답. (P2-1 보강 — 이전 표현은 "DB validators" 라고 잘못 적었음. `MinValueValidator` / `MaxValueValidator` 는 Django validation hook 일 뿐 DB constraint 가 아니라 raw SQL / `.save()` 직접 호출은 우회 가능. 실제 DB-level guard 는 `CheckConstraint` 가 별도로 필요.)

- **DB 차원** (raw SQL / `.save()` 우회 방지):
  - `PositiveSmallIntegerField` — 음수·overflow 차단.
  - **`Meta.constraints = [CheckConstraint(...)]` 두 개** — `max_iterations BETWEEN 1 AND 12`, `max_low_relevance_retrieves BETWEEN 1 AND 10`. PostgreSQL 이 INSERT/UPDATE 시점에 강제, 위반 시 `IntegrityError`.
- **Form 차원** (사용자 입력 차단):
  - `ModelForm` 이 `MinValueValidator` / `MaxValueValidator` 를 상속 → POST 검증 시 한국어 에러 메시지.
- **Runtime 차원** (DB constraint 우회 / 마이그레이션 누락 시 마지막 안전망):
  - `load_runtime_settings()` 가 DB 값 읽고 **sanity check** (max_iterations 가 0 이하 / 비정상 값) → `_DEFAULTS` 폴백 + `logger.warning`. CheckConstraint 가 정상 환경에선 raw SQL 도 잡지만, **마이그레이션 누락된 환경** (CheckConstraint 적용 전 row) / **외부 데이터 dump 로 비정상 row import** / **DB 자체 손상** 등 극단 상황에서도 runtime 깨지지 않게.
  - 테스트는 manager 를 **mock 으로 대체**해 비정상 dataclass 를 반환시키거나, `AgentSettings` 객체 직접 생성 후 manager 만 patch (실제 DB INSERT 안 함) — CheckConstraint 가 정상 INSERT/UPDATE 를 차단하는 환경에서 sanity 분기 코드 자체를 격리 검증 (P3-2 보강).

triple guard 의 의도: BO 폼 → DB constraint → runtime sanity 어느 한 단이 뚫려도 다음 단이 잡음.

### 3. 즉시 반영 vs 재시작 — **즉시 반영**

설계 §7-3 세 번째 질문에 대한 답.

- `load_runtime_settings()` 는 **매 `run_agent` 호출마다** DB SELECT.
- agent 호출 빈도 (사용자당 분당 1~2 회) 대비 SELECT 비용 무시 가능 (< 1ms).
- TTL 캐시 도입은 incident 대응 가치 약화 — "BO 에서 끔" 과 "실제 끔" 사이 5~30초 윈도우가 incident 시 부담.
- agent 호출 중 동시에 settings 가 바뀌어도 race 는 없음 — 한 호출 안에서는 시작 시점의 snapshot 사용.

### 4. agent disabled 시 처리 — **single_shot 폴백** (UNSUPPORTED 아님)

설계 §7-3 네 번째 질문에 대한 답.

옵션:
- (A) single_shot 폴백 — 사용자에게 답변은 가지만 quality 낮을 수 있음.
- (B) UNSUPPORTED 카피 ("이 질문은 일시적으로 답변할 수 없습니다") — 사용자 경험 나쁨.

본 plan 은 **(A)**. 이유:
- 운영자가 agent 를 끄는 시점 = incident 대응 시점 — 사용자에겐 답변이 안 가는 것보다 quality 낮은 답변이 가는 게 나음.
- single_shot 도 RAG retrieval 이 동작하므로 자료 기반 답변은 여전히 가능.
- Phase 7-2 의 single_shot 폴백 패턴 재사용 — 코드 / 멘탈 모델 일관.
- 후속 Phase 에서 사용자별 정책이 필요하면 그때 (B) 도 옵션으로 도입 가능.

### 5. tool catalog 의 노출 수준 — **읽기 전용, 메타만**

설계 §7-4 권장 방향: "tool catalog 는 먼저 읽기 전용으로 공개. 쓰기 가능한 tool 제어는 후속 Phase".

본 plan 노출 항목:
- `name` (예: `retrieve_documents`)
- `description` (Tool dataclass 의 description 필드 — 한국어 한 줄)
- `input_schema 모드` ("schema" / "raw" — `input_schema` 가 None 이면 raw)
- `failure_check 등록 여부` (운영자가 retrieve 의 low_relevance 가드 인지하게)

비공개 항목:
- callable / summarize 함수 이름·코드 — 보안·혼란 우려.
- 호출 빈도·실패율 — Phase 8-4 또는 후속 (TokenUsage purpose 데이터 활용).

### 6. AgentSettings singleton 의 row 보장 — **데이터 마이그레이션**

`get_solo()` 만으로도 첫 호출 시 row 생성되지만, 새 환경에서 **첫 BO 페이지 GET 이전에** runtime 이 settings 를 조회할 수 있음 (예: 첫 채팅 요청). row 가 없을 때 `_DEFAULTS` 폴백이지만, BO 페이지에서 "처음 본 값과 첫 GET 후 본 값이 다름" 같은 혼란 방지를 위해 **마이그레이션에 RunPython 으로 초기 1행 명시 생성**.

```python
def _create_default_settings(apps, schema_editor):
    AgentSettings = apps.get_model('chat', 'AgentSettings')
    AgentSettings.objects.get_or_create(pk=1, defaults={
        'enabled': True, 'max_iterations': 6, 'max_low_relevance_retrieves': 3,
    })
```

이 한 단계로 모든 환경 (dev / test / production) 이 동일 초기 상태.

### 사용자-facing smoke

8-3 의 사용자-facing 변화는 **agent disabled 시 single_shot 폴백** 한 가지. 운영 시나리오 검증:

1. 정상 상태 (default `enabled=True, max_iterations=6, max_low_relevance_retrieves=3`):
   - 비교형 질문 (예: `본인 결혼 경조금이랑 자녀 결혼 경조금 비교해줘`) → agent route → 정상 답변. 서버 로그에 `agent step 0~N`, `agent 실행: status=ok`.

2. BO `/bo/agent/` 진입:
   - 폼에 default 3 값 (enabled=True / max_iter=6 / max_low_rel=3).
   - tool catalog 에 `retrieve_documents / find_canonical_qa / run_workflow` 3개 노출 + 각 description / 모드.
   - 최근 7일 카드: `agent_step` 카운트 / `agent_final` 카운트 (Phase 8-2 데이터 활용).

3. `enabled=False` 저장 → 같은 비교형 질문:
   - 라우터는 여전히 agent 로 결정 (RouterRule 매칭 그대로).
   - `agent_node` 진입 시 settings.enabled=False 감지 → `single_shot_node(state)` 호출.
   - 서버 로그: `INFO chat.graph.nodes.agent: agent 비활성 — single_shot 폴백`.
   - 사용자에게 답변은 옴 (single_shot 답변).
   - TokenUsage 신규 row (P3-1 보강):
     - `agent_step` / `agent_final` **0** (agent runtime 진입 안 함).
     - `single_shot_answer` **+1**.
     - `query_rewriter` — **세션 history 가 누적된 경우 +1** (single_shot pipeline 도 history 가 있으면 rewriter 호출). 폴백 검증을 정확히 하려면 **fresh no-history 세션** 에서 한 번 더 검증해 `query_rewriter=0` 가 도달하는지 확인.
   - **권장**: `enabled=False` 검증은 fresh 세션 (rewriter 미발생) + 같은 세션 후속 (rewriter 발생) 두 번 — 폴백 자체는 history 와 무관하게 항상 동작해야 함.

4. `max_iterations=2` 저장 → 비교형 질문 (다중 retrieve 필요):

   - 서버 로그: `agent step 0`, `agent step 1`, `agent 실행: status=not_found`.

5. POST `max_iterations=99` (validator 위반):

## 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| `load_runtime_settings()` 가 매 호출마다 DB SELECT — 부하 | TokenUsage / RouterRule 도 매 호출마다 SELECT. agent 호출 빈도 (분당 1~2 회) 대비 SELECT 1회 추가는 무시 가능. 후속 부하 모니터링에서 문제 시 5초 TTL 캐시 도입 검토. |
| AgentSettings row 가 없는 환경에서 첫 페이지 GET 까지 `_DEFAULTS` 폴백 | Step 1 의 RunPython 으로 마이그레이션 시점에 1행 보장 → 모든 환경 동일 초기. 만약 마이그레이션 후 row 가 삭제돼도 (Django admin) `_DEFAULTS` 가 잡음. |
| BO 페이지 운영자 오조작으로 enabled=False 영구 유지 | updated_at 표시 + "마지막 변경: N분 전" 안내. audit 로그는 후속 Phase. |
| singleton 패턴이 race 발생 / shell · admin 으로 추가 row 생성 | (a) `get_solo()` 의 `get_or_create(pk=1)` 가 transaction-safe — race 시에도 두 번째 트랜잭션이 PK 충돌로 실패. (b) `AgentSettings.save()` override 가 `self.pk=1` 강제 — **결과 동작** (P3-2 보강): `objects.create(pk=2, ...)` 는 내부적으로 `save(force_insert=True)` 호출 → save override 에서 pk=1 강제 → INSERT INTO ... pk=1 시도 → 기존 row 존재 시 `IntegrityError` (UPDATE 가 아님). 첫 row 가 없는 환경에선 단순히 pk=1 으로 INSERT. `obj.save()` (default force_insert=False) 는 pk=1 으로 UPDATE 또는 INSERT 자동 분기. **결론**: 어떤 호출 경로로도 pk≠1 row 가 만들어지지 않음. extra row 0 보장. (c) Django admin 등록 안 함 (BO 전용). |
| 기존 `react.py` 호출자가 `DEFAULT_MAX_ITERATIONS` 를 import 해서 사용 | 본 plan 은 상수 보존 — 호출자 무영향. settings 우선 적용은 `run_agent` 내부 로직 변경만. |
| BO `Agent` 메뉴 추가가 기존 nav 디자인을 깨뜨림 | base.html 의 기존 nav-item 패턴 그대로 (Files / Q&A / Prompts / Router Rules) 따라 5번째 항목으로 추가. CSS 변경 0. |
| Form validator 메시지가 한국어 / 영어 혼재 | Django 기본 메시지가 한국어 (`LANGUAGE_CODE='ko-kr'`) 인지 확인. 부족하면 `error_messages` 로 명시. |
| Phase 4-2 RouterRule 페이지와 의미 충돌 | RouterRule = 라우팅 규칙 (질문 → route 결정). AgentSettings = agent runtime 동작 한도 / on-off. 완전 다른 layer — UI 에서 두 페이지 분리 명확화. |
