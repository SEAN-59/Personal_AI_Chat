# 0.4.0 Phase 9 — 안정화 및 배포 준비 개발 플랜

## Context

Phase 8-6 머지 직후 (2026-04-29) 시점:

- 로드맵 (`resources/plans/0.4.0_랭그래프_리팩토링_로드맵.md` §3) 의 **본 Phase 8 의도는 "안정화 + 배포 준비"** 였지만, 실제 개발은 Phase 8 을 "agent 운영화" 로 재정의해 8-1~8-6 (6 PR) 진행. 따라서 **로드맵의 Phase 8 본 의도는 미진행** — 별 phase (Phase 9) 로 분리해 본 plan 에서 처리.
- Phase 0 (기준선 확보) 도 별 dev log / plan 없이 자연 진행. 대표 질문셋 / 회귀 체크리스트 미정리.
- 504 테스트 그린 / 6 milestone closed (8-1 ~ 8-6) — agent 영역의 운영자 control plane 완성. 다만 release 전 회귀 / 라우팅 / 프롬프트 / 마이그레이션 / 배포 체크는 별 작업.

Phase 9 의 목표: **release 직전 안정화 작업** 으로 (1) 회귀 검증 자동화, (2) 라우팅 정합 점검, (3) 프롬프트 / 문서 정리, (4) 배포 체크리스트 작성. 코드 변경 최소 / 검증 / 문서 중심.

> 본 plan 은 로드맵 §3 의 Phase 8 (안정화 + 배포 준비) 항목에 직접 매핑. 8-1~8-6 의 polish backlog 일괄 처리는 본 phase 의 책임이 아님 (별 milestone).

### 1. 회귀 체크리스트 + 대표 질문셋

- `resources/documents/2026-04-XX-phase9-stabilization.md` 안에 **대표 질문셋 + 기대 라우트 / 답변 형태** 정리. 질문은 **실제 라우터 (RouterRule + 코드 키워드 fallback) 계층 + 현재 workflow 기능 형태** 정합:
  - **single_shot** ×3 — `WORKFLOW_KEYWORDS` / `AGENT_KEYWORDS` 미매치 자료 기반 질문 (예: `경조사 규정 알려줘` / `복리후생 항목별로 정리해줘` / `회사 휴가 종류 알려줘`). **`퇴직금` / `계산` 등 키워드는 코드 fallback 의 `WORKFLOW_KEYWORDS` 매치라 single_shot 으로 안 떨어짐 — P2-1 보강** (운영 RouterRule 도 그 키워드는 workflow_key='' 로 잡힐 수 있어 graph 단에서 single_shot 폴백되지만, 본 e2e 테스트의 라우터 기대값은 그대로 `workflow`).
  - **workflow** ×3 — 현재 지원하는 형태로 (P2-2 보강):
    - date_calculation: `2025-01-01부터 2025-04-11까지 며칠?` (시작/종료일 형태 — extractor prompt 가 상대/파생 날짜는 거부).
    - amount_calculation: `100 + 50 합계는?` (만 단위 배율 적용 안 함 — 단순 숫자 합계).
    - table_lookup: `복리후생 규정에서 본인 결혼 경조금은?` (표 PDF 가 운영 환경에 있을 때만).
  - **agent** ×3 — 비교형 1 (`본인 결혼 경조금이랑 자녀 결혼 경조금 비교해줘`) / 다중 retrieve 1 (`복리후생 규정과 취업규칙의 휴가 항목 비교`) / no-info 1 (`우주여행 비용 비교` — Phase 8-4 가드 검증).
  - **edge cases** ×3 — 빈 질문 / 매우 긴 질문 / 인사 (`안녕하세요` — 자료 없음 응답 정합, Phase 9 시점 정책 그대로).
- 각 질문에 대해:
  - 기대 라우트 (single_shot / workflow / agent).
  - 기대 `workflow_key` (workflow 의 경우 — RouterRule 미등록 환경에선 `''` 로 single_shot 폴백, 등록 환경에선 매핑된 key).
  - 기대 응답 형태 (자료 기반 / 계산 / NOT_FOUND 카피 / 캐시 히트).
  - 기대 sources 노출 여부 / 개수.
  - 기대 TokenUsage purpose 분포.

### 2. 3분류 라우팅 정합 검증 (자동 smoke 신규)

- `chat/tests/test_routing_e2e.py` **신규**:
  - `route_question(<대표 질문>)` 가 의도된 route 로 결정되는지 e2e 검증.
  - **세 layer 모두 cover** (P2-1 보강):
    1. **DB RouterRule layer** — `setUp` 에서 `RouterRule.objects.create(...)` fixture 명시 생성 (clean DB 라 fixture 없으면 코드 키워드 fallback 으로 떨어져 workflow_key 가 빈 문자열). workflow 라우팅 + workflow_key 매핑 검증 (e.g. `pattern='며칠'`, `route='workflow'`, `workflow_key='date_calculation'` row).
    2. **코드 키워드 fallback layer** — RouterRule 미등록 시 `WORKFLOW_KEYWORDS` / `AGENT_KEYWORDS` 매치 → 라우트 결정 (workflow_key='' 로 떨어짐).
    3. **default layer** — 어떤 키워드도 매치 안 되는 일반 자료 질문 → single_shot.
  - 9 cases — single_shot 3 / workflow 3 (RouterRule fixture + workflow_key 검증) / agent 3.
- `chat/tests/test_pipeline_smoke.py` **신규**:
  - `run_chat_graph(<대표 질문>, history=[])` end-to-end 호출 — **노드 자체 mock 방식 채택** (검증 의도가 "라우터 → 노드 진입 정합" 이라 노드 안의 LLM/embedding 동작은 별 layer 의 단위 테스트가 cover):
    - `chat.graph.nodes.single_shot.single_shot_node` / `chat.graph.nodes.workflow.workflow_node` / `chat.graph.nodes.agent.agent_node` 를 patch.
    - patch target 은 graph 가 import 한 binding (`chat.graph.app.single_shot_node` / `workflow_node` / `agent_node`) — `chat.graph.app.py` 의 import 문 기준으로 결정.
  - **`_compiled_graph` lru_cache 비우기 계약** (P2 보강 — `chat.graph.app._compiled_graph` 는 `lru_cache` 적용 함수라 patch 전 cached graph 가 원래 node binding 을 들고 있어 mock 무시. 기존 `test_graph_agent_wiring.py` 가 같은 패턴 사용):

  - 9 경로 → 9 cases. mock 으로 모든 외부 호출 차단 — 실제 OpenAI / 임베딩 검색 / reranker 호출 0회.

## 핵심 설계 결정

### 1. Phase 9 vs Phase 8-7 / 8-8 — **새 milestone**

옵션:
- (A) Phase 8-7 / 8-8 등으로 Phase 8 milestone 안에 누적.
- (B) **Phase 9 신규 milestone** — 채택 권장.

**(B)** 이유:
- Phase 8 의 의미 = "agent 운영화" (6 PR 로 완성). 안정화 / 배포 준비는 의미가 다름 (cross-cutting).
- git-flow.md 정책 ("모든 이슈 닫히면 마일스톤도 닫는다") 정합 — Phase 8 닫고 Phase 9 시작.
- 후속 backlog 도 Phase 9 안에 누적 가능 (안정화 = 운영 검증 의미).

다만 사용자가 (A) 원하면 본 plan 의 milestone 을 그대로 Phase 8 안에 두는 옵션도 무방 — milestone 명명은 행정 결정 영역.

### 2. 코드 변경 최소화 — 검증 + 문서 중심

본 phase 의 본 의도가 "release 전 안정화" 라 **코드 변경 0** 이 이상적. 신규 추가는:
- e2e 라우팅 / 파이프라인 smoke 테스트 (검증용).
- 문서 / 체크리스트.

기능 추가 / 수정 / refactor 는 본 phase 범위 밖 — 발견된 잡음은 별 backlog 로.

### 3. 회귀 체크리스트의 형식

옵션:
- (A) Markdown 표 — 사람이 수동 검증.
- (B) **자동 smoke + Markdown 보조** — 채택.

**(B)** 이유:
- 자동 smoke (라우팅 e2e / 파이프라인 mock) 가 회귀의 1차 가드.
- Markdown 표는 release 직전 사용자-facing 수동 검증 (브라우저 클릭) 의 가이드.
- 두 layer 합쳐 "코드 + 사용자 경험" 모두 cover.

### 4. CHANGELOG vs release note

옵션:
- (A) `CHANGELOG.md` 신규 — 표준 형식 (Keep a Changelog).
- (B) GitHub release note 만 — release tag 시점에 작성.

**(A)** 채택 — repo 안에 history 보존 + GitHub release note 가 본 파일 인용. version 별 변화 추적이 git 외부에서도 가능.

### 5. 배포 체크리스트의 위치

- 본 phase 의 dev log (`resources/documents/2026-04-XX-phase9-stabilization.md`) 안에 §배포 섹션으로.
- 별 `DEPLOYMENT.md` 신규는 over-engineering — 0.3.x 시점부터 dev log 패턴 일관.
- 후속 release (2.1.0 등) 에서 같은 dev log 패턴 재사용.
