# 2.0.0 Phase 3 — 기존 코드 개편 개발 플랜

## Context

Phase 2 에서 `chat/views/message.py` → LangGraph → `single_shot node` → `answer_question()` 경로가 생겼지만, 실제 single-shot 로직은 여전히 `chat/services/query_pipeline.py` 하나의 큰 함수에 orchestration 과 business logic 이 뒤섞인 상태다. 이 상태로 Phase 4 (router 실제 분기), Phase 5~6 (workflow), Phase 7 (agent) 를 얹으면 node 안이나 service 곳곳에서 같은 흐름이 중복되고, single-shot 의 helper 들을 workflow 가 재사용하기도 어렵다.

Phase 3 의 목표는 **사용자 관점 동작은 동일하게 유지한 채, `answer_question()` 을 7개 단계로 분해하고 `chat/services/single_shot/` 패키지로 이관 + LangGraph 진입점을 이 패키지로 직접 연결하는 것**. 동시에 workflow 자리(`chat/workflows/`) 를 미리 만들어 두고, Phase 2 에서 안전장치로 남겼던 `USE_LANGGRAPH_PIPELINE` feature flag 를 제거해 direct 경로를 완전히 걷어낸다.

핵심 제약:
- 응답 품질·JSON 포맷·히스토리 동작 **회귀 0** — Phase 1/2 때와 동일 기준
- 검색·재정렬·프롬프트 조립·OpenAI 호출 **로직 자체는 변경 금지**. 위치만 옮김
- 파일만 늘리는 분해는 금지 — "역할이 분명할 때만 분리" (설계 §13-1)
- `files/` 앱, BO, 문서 처리 파이프라인은 **손대지 않는다**

## 핵심 설계 결정

### 1. `query_pipeline.py` 의 운명: 얇은 shim 으로 존치 → PR 마지막 커밋에서 삭제
`answer_question`, `QueryResult`, `QueryPipelineError` 세 심볼을 유지하되 내부는 `single_shot.pipeline.run_single_shot` / `single_shot.types` 로 재export. 이유:
- 외부(`chat/graph/nodes/single_shot.py`, view)에서 `chat.services.query_pipeline` 을 import 하는 습관이 이미 있어 완전 삭제는 cross-commit 불일치를 부름.
- Phase 3 내에서 node·view 가 새 경로로 전환된 뒤, Phase 3 마지막 커밋에서 `query_pipeline.py` 를 **삭제**한다. 이 PR 안에서 최종 상태는 "삭제됨".
- 즉 이 파일은 "리팩토링 도중에만 존재하는 중간 다리" — 최종 트리엔 안 남는다.

### 2. Types 모듈 분리
`QueryResult` 와 `QueryPipelineError` 를 `chat/services/single_shot/types.py` 로 이동. 이유:
- `pipeline.py` 에서 이들 타입을 반환/raise 하고, `nodes/single_shot.py`, `graph/app.py`, `graph/state.py`, view 가 모두 import 함 — 순환 import 피하려면 단일 장소에 모으는 게 맞음.
- Phase 2 에서 `graph/state.py` 가 `chat.services.query_pipeline` 의 `QueryResult` 를 import 했는데, Phase 3 에서는 `chat.services.single_shot.types` 를 import 하도록 교체.

### 3. helper 입출력 시그니처: "질문 하나, dataclass 하나씩"
helper 간 전달은 기본형 또는 기존 dataclass 로만. side effect(DB 저장·로그) 는 오직 `postprocess` 안에서만 일어남 — 다른 helper 는 pure.

| helper | in | out |
|---|---|---|
| `retrieval.retrieve_documents` | `question: str` | `list[ChunkHit]` (재정렬 완료) |
| `qa_cache.find_canonical_qa` | `question: str` | `list[QAHit]` |
| `qa_cache.resolve_cache_hit` | `qa_hits` | `Optional[QueryResult]` (히트면 완성된 결과) |
| `prompting.build_messages` | `question, chunk_hits, qa_hits, history` | `list[dict]` |
| `llm.run_chat_completion` | `messages` | `(reply: str, usage, model: str)` (OpenAI usage 타입 그대로) |
| `postprocess.classify_reply` | `reply` | `(is_no_info, is_casual)` |
| `postprocess.record_token_usage` | `model, usage` | `None` (DB 저장) |
| `postprocess.persist_chat_log` | `question, reply, chunk_hits` | `Optional[int]` chat_log_id |
| `postprocess.build_sources` | `chunk_hits` | `list[dict]` |
| `pipeline.run_single_shot` | `question, history` | `QueryResult` (조합) |

`prompting.build_messages` 는 기존 `chat.services.prompt_builder.build_messages` 를 **얇게 감싸는 래퍼**로 시작. 이유: Phase 1 에서 이미 안정화된 builder 를 Phase 3 의 역할 재정렬과 섞지 않는다. 이후 phase 에서 workflow/agent 전용 builder 가 추가될 때 이 래퍼 위치가 자연스러운 확장 지점이 됨.

### 4. flag 제거 타이밍
Phase 3 마지막 리팩토링 커밋(Step 8)에서 깨끗이 제거:
- `AI_Chat/settings.py`: `USE_LANGGRAPH_PIPELINE` 설정 삭제
- `env.example.txt` / `env.prod.example.txt`: 관련 주석 삭제
- `chat/views/message.py`: `if settings.USE_LANGGRAPH_PIPELINE:` 분기 삭제, graph 경로만 남김
- Phase 2 dev log 의 Phase 3 handover 항목과 일관

### 5. workflows/ 골격은 텅 빈 `__init__.py` 만
각 `__init__.py` 에 한 줄 docstring(`"""Phase 5~6 에서 구현 예정."""`)만 두고 실 코드 없음. 디렉토리만 먼저 만들어 Phase 5 PR 이 바로 파일을 얹을 수 있게.

### 6. 공통 규칙 명문화 위치
`chat/services/single_shot/__init__.py` 의 module docstring. 내용:
- **history**: 세션 히스토리는 view(`message.py`) 에서만 load/save. service·graph 는 건드리지 않음.
- **error**: service 내부는 `QueryPipelineError` 만 raise. 다른 예외는 상위로 올림 → 노드가 `state.error` 로 포획 → view 가 502.
- **token usage**: OpenAI 호출 직후 `postprocess.record_token_usage` 에서 기록. 다른 위치에서 중복 기록 금지.
- **chat log**: 오직 `postprocess.persist_chat_log` 에서. 캐시 히트 시 저장하지 않음(기존 규칙 유지).
- **sources**: `postprocess.build_sources` 에서만 구성. chunk 중복 제거 규칙 포함.
- **잡담·no-info 분류**: `postprocess.classify_reply` 결과로 sources / chat log 억제.

이 docstring 이 Phase 5~7 에서 같은 규칙을 적용하는 기준점이 된다.
