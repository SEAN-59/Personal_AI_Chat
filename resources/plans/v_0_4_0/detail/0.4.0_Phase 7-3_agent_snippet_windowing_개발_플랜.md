## Context

## 핵심 설계 결정

### 1. Tool API 는 그대로 — `_retrieve_callable` 반환을 dict 로 감싸기

현재 (7-2):

```python
def _retrieve_callable(arguments):
    return _retrieve(arguments['query'])           # list[Chunk]

def _summarize_retrieve(hits):                     # signature: Callable[[Any], str]
    ...
```

`Tool.summarize` 의 시그니처를 `Callable[[Any, Mapping], str]` 로 바꾸면 가장 깔끔하지만, 다른 도구(`find_canonical_qa`, `run_workflow`)와 향후 추가될 도구 모두에 영향. 본 PR 은 **retrieve 한정 격리 변경** 으로 처리:

```python
def _retrieve_callable(arguments):
    query = arguments['query']
    return {
        'query': query,
        'hits': _retrieve(query),
    }

def _summarize_retrieve(result):                   # 여전히 Callable[[Any], str]
    query = (result or {}).get('query') or ''
    hits = (result or {}).get('hits') or []
    ...
```

장점:
- Tool 데이터타입 / `tools.call` 흐름 / 다른 도구 영향 0.
- `result` 는 internal 형태이고 외부에 노출되는 건 `Observation.summary` (문자열) 뿐.

Trade-off (수용):
- 형태가 retrieve 만 dict 라 일관성 떨어짐. 하지만 본 우회는 tools_builtin.py 한 모듈 안에서만 보이므로 격리됨. 후속 Phase 에서 다른 도구도 query-aware summarize 가 필요해지면 그때 Tool API 시그니처 변경을 한 번에 진행.

### 2. `_focus_window` 알고리즘

```python
import string

_KEYWORD_MIN_LEN = 2
# 토큰 양 끝에 흔히 붙는 문장부호 제거용. ASCII punctuation + 한국어 콤마/물음표 등.
_TOKEN_STRIP_CHARS = string.punctuation + '·、，。？！'


def _tokenize_query(query: str) -> list[str]:
    """Query 를 윈도우 매칭용 토큰으로 분리.

    - 공백 split.
    - 토큰 양 끝 punctuation strip (`결혼?` → `결혼`, `"경조금"` → `경조금`).
    - 길이 ≥ 2 만 유지 (1자 조사/어미/stopword 제거 — `는`, `이`, `을`).
    - **길이 내림차순 정렬** — 긴 토큰일수록 도메인 키워드일 확률이 높음 (예:
      `경조금`(3) > `비교`(2)). 매치 시 긴 토큰 위치 우선 → 일반 토큰이 청크
      앞부분에 우연히 걸려 관련 없는 윈도우를 고르는 케이스 차단.

    한국어 형태소 분석기 (KoNLPy / Mecab) 도입은 의존성 비용 큼. 운영 데이터에서
    부족이 입증되면 후속 Phase 에서 검토.
    """
    tokens = []
    for raw in query.split():
        cleaned = raw.strip(_TOKEN_STRIP_CHARS)
        if len(cleaned) >= _KEYWORD_MIN_LEN:
            tokens.append(cleaned)
    # stable sort by length desc — Python 의 sort 는 안정 정렬이라 같은 길이는 입력 순서 유지.
    tokens.sort(key=len, reverse=True)
    return tokens


def _focus_window(content: str, query: str, *, length: int) -> str:
    """Query 키워드 매치 위치 주변 ±윈도우. 미매치면 청크 첫 N자 fallback.

    윈도우 정책: 매치 위치 기준 앞 1/4 + 뒤 3/4 (forward-bias). 표 행은 보통
    매치 위치 다음에 값/단위가 오므로 뒤를 더 본다. 매치가 청크 매우 앞부분
    (< length//4) 이면 `start = max(0, earliest - length//4)` 가 자연스럽게 0
    으로 클램프돼 첫 N자 출력 = 7-2 fallback 과 byte-identical.

    의도적으로 `earliest < length` 강제 분기를 두지 않는다. 이전 초안은 "매치가
    첫 length 안이면 무조건 첫 N자" 로 강제했는데, 이건 키워드는 350자에 있는데
    값은 450자에 있는 흔한 표 패턴 — 즉 7-3 의 본 목적인 401자+ 답 노출을 일부
    막아버린다. forward-bias 한 줄기로 처리하면 keyword 위치별로 자연스럽게:
        earliest < length//4   → start=0     (7-2 byte-identical)
        length//4 ≤ earliest   → 매치 주변 ±윈도우 (7-3 의 본 가치)
    """
    if not content:
        return ''
    if len(content) <= length:
        return content
    if not query:
        return content[:length] + '…'

    tokens = _tokenize_query(query)
    if not tokens:
        return content[:length] + '…'

    # 긴 토큰부터 매치. 정렬 덕분에 첫 매치 = 가장 긴 매치된 토큰의 위치.
    # 짧은 토큰("비교") 이 청크 앞부분에 우연히 걸려 관련 없는 윈도우를 고르지 않게.
    lower = content.lower()
    earliest = -1
    for token in tokens:
        idx = lower.find(token.lower())
        if idx >= 0:
            earliest = idx
            break

    if earliest < 0:
        # 미매치 — 청크 첫 N자 fallback (7-2 동작과 동일).
        return content[:length] + '…'

    # forward-bias 윈도우. 앞 1/4 + 뒤 3/4.
    pre = length // 4
    start = max(0, earliest - pre)         # 매치가 청크 앞쪽이면 자연 클램프 → 0
    end = min(len(content), start + length)
    # content 끝에 닿으면 start 를 뒤로 당겨 윈도우 길이 보존.
    start = max(0, end - length)

    snippet = content[start:end]
    prefix = '…' if start > 0 else ''
    suffix = '…' if end < len(content) else ''
    return prefix + snippet + suffix
```

설계 요약:

- **토큰화** (P2-3 보강):
  - 공백 split.
  - 양 끝 punctuation strip — `결혼?` / `"경조금"` 도 매치 가능.
  - 길이≥2 필터 — 1자 조사/어미/stopword 제거.
  - **길이 내림차순 정렬** — 긴 토큰부터 매치 시도. `결혼 경조금 비교` 같은 query 에서 `경조금`(3자) 이 청크 600자 위치, `비교`(2자) 가 청크 50자 위치라면 `경조금` 위치 (600) 가 윈도우 중심이 됨. 짧은 일반 토큰 ("비교", "있는", "하는", "문서") 이 청크 앞부분에 걸려 관련 없는 윈도우를 고르는 회귀 차단.
- **case-insensitive**: `.lower()` 사용. 영문 키워드 대응.
- **forward-bias 윈도우 단일 정책** (P2-1 재보강): 매치 위치 기준 앞 1/4 + 뒤 3/4. `start = max(0, earliest - length//4)` 의 자연 클램프 덕분에 매치가 청크 매우 앞 (`< length//4`) 이면 자동으로 `start=0` → 첫 N자 출력 = 7-2 fallback **byte-identical**. 별도 `if earliest < length` 강제 분기 없음 — 그건 키워드는 350자, 값은 450자 같은 흔한 표 패턴에서 본 목적을 깨뜨린다 (P2 추가 지적).
- **forward-bias 의 의미**: 표 행은 보통 `본인 결혼 100만원` 처럼 매치 위치 (`결혼`) 다음에 값/단위가 옴. 뒤쪽을 더 보여주면 매치 키워드 + 값/단위/조건이 같은 윈도우에 같이 들어옴.
- **fallback**: 미매치 시 청크 첫 N자 + `…` — 7-2 와 동일.
- **truncation marker**: `…` 로 잘림 표시. LLM 이 "더 있다" 인지하고 query 다듬어 retry 가능 (7-2 프롬프트 가이드와 호응).
- **edge cases**:
  - 빈 content → 빈 문자열.
  - content 가 length 보다 짧음 → 그대로 반환 (잘림 표시 없음).
  - 빈 query → 첫 N자 + `…`.
  - 토큰이 모두 1자 → 첫 N자 fallback.
  - 매치 위치가 끝 근처 → start/end 재조정으로 윈도우 길이 보존.

### 3. `_summarize_retrieve` 통합

```python
def _summarize_retrieve(result):
    """top N 청크의 출처 + query 키워드 주변 윈도우. (Phase 7-3)

    Phase 7-2 까지는 청크 첫 N자만 노출했지만, 답이 N+1 자 이후에 있으면 LLM
    이 못 보는 회귀가 있어 query 매치 위치 주변 윈도우로 전환. 키워드 매치
    미발견 시 fallback 은 첫 N자 (= 7-2 동작과 동일).
    """
    query = (result or {}).get('query') or ''
    hits = (result or {}).get('hits') or []
    if not hits:
        return '검색 결과 없음 (0건)'

    parts = [f'{len(hits)}건 검색됨:']
    for idx, hit in enumerate(hits[:_RETRIEVE_TOP_N], start=1):
        name = getattr(hit, 'document_name', None) or '(출처 미상)'
        content = (getattr(hit, 'content', '') or '').replace('\n', ' ').strip()
        snippet = _focus_window(content, query, length=_RETRIEVE_SNIPPET_LEN)
        parts.append(f'[{idx}] {name}: "{snippet}"')
    if len(hits) > _RETRIEVE_TOP_N:
        parts.append(f'(이하 {len(hits) - _RETRIEVE_TOP_N}건 생략)')
    return ' '.join(parts)
```

기존 테스트들이 `_retrieve` (모듈 함수) 를 mock 하므로 `_retrieve_callable` 의 dict wrapping 은 mock 대상과 무관 → 호환성 유지.

### 4. 회귀 가드

회귀 0 의 정확한 정의 (P2-1 + P2 추가 지적 반영, `length=400` 기준):

| earliest 위치 | 7-2 출력 | 7-3 출력 | 동등성 |
|---|---|---|---|
| `0 ≤ earliest < length//4` (예: 0~99) | 청크 0~400자 | **청크 0~400자 (자연 클램프)** | byte-identical (회귀 0) |
| `length//4 ≤ earliest < length` (100~399) | 청크 0~400자 | 매치 주변 윈도우 (예: 250~649) | 출력 다름. 값이 length 너머 (~450 등) 에 있어도 노출 가능 — **본 목적 일부** |
| `earliest ≥ length` (400+) | 청크 0~400자 (답 못 봄) | 매치 주변 윈도우 (예: 500~899) | 출력 다름. **본 목적** — 이전 못 보던 답 노출 |
| 매치 미발견 | 청크 0~400자 | 청크 0~400자 + `…` | byte-identical (회귀 0) |

핵심: **출력 텍스트 byte-identical 회귀 0** 은 매치가 매우 앞 (`< length//4`) / 미발견 두 케이스에서만 보장. 그 외 구간에서는 출력이 의도적으로 달라지지만 그건 답변 능력 향상이 본 목적이라 의도된 변화. 이전 초안의 "earliest < length 면 무조건 첫 N자" 는 키워드 350자 + 값 450자 같은 케이스에서 본 목적을 깨뜨려 폐기.

추가 가드:
- summary 길이 cap (1500) 그대로 — 한 턴 컨텍스트 budget 변화 없음.
- Tool API signature 변경 0 — 다른 도구 / 외부 코드 영향 없음.
- 비-retrieve 도구 (`find_canonical_qa`, `run_workflow`) summary 변경 0.
- `_retrieve` 함수 자체 호출 수 / 인자 변경 0 — retrieval 결과는 동일.

  - **(P2 추가 지적 — 경계 케이스)** 키워드는 length 안 (예: 350자), 값은 length 너머 (예: 450자) → forward-bias 윈도우 발동 (예: 250~649) 으로 값이 윈도우에 포함. 7-2 가 못 보던 답이 노출되는 본 목적의 케이스.
  - **(windowing 본 무대)** 매치 위치가 length 너머 (예: 600자) → 윈도우가 매치 주변 (forward-bias 앞 1/4 + 뒤 3/4).
  - 매치가 content 끝 근처 (예: content 길이 - 50자 위치) → 윈도우 끝이 content 끝에 달라붙고 prefix `…` 만 / suffix `…` 없음.
  - 매치가 length 너머지만 시작 근처 (length 바로 뒤, 예: 410자) → start 클램프 정상 동작.
  - 미매치 (query 토큰이 모두 content 에 없음) → 첫 N자 + `…`.
  - case-insensitive 매치 (`MAX` query → `max` content 매치).
  - **(P2-3)** 긴 토큰 우선: query "비교 결혼 경조금" / 청크에 "비교" 가 50자 위치 + "경조금" 이 600자 위치 → 윈도우 중심이 600 (`경조금`).
  - **(P2-3)** punctuation strip: query "결혼?" → "결혼" 토큰으로 매치.
- 아직 `_summarize_retrieve` 통합 안 함 — helper 격리 검증.
- **커밋**: `feat: Add query-focused snippet windowing helper (#47)`

## 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| 한국어 토큰화가 공백 split 만으로 부족 (예: "결혼경조금" 처럼 붙여쓴 query) | 한국어 형태소 분석기 (KoNLPy / Mecab) 도입은 의존성 비용 큼. 공백 split + 길이≥2 으로 시작. 운영에서 불충분 입증되면 후속 Phase 에서 검토. 프롬프트의 "query 다듬어 retry" 가이드가 안전망 |
| query 의 stopword (예: "있는", "하는") / 짧은 일반 토큰 (예: "비교", "문서") 이 청크 앞부분에 우연히 걸려 관련 없는 윈도우를 고름 | (P2-3) 길이≥2 필터로 1자 stopword 제거 + **길이 내림차순 정렬로 긴 토큰부터 매치 시도**. `결혼 경조금 비교` query 에서 청크에 `경조금`(3자) 이 있으면 그 위치가 윈도우 중심이 됨 — `비교`(2자) 가 청크 앞부분에 있어도 무시. 더 정교한 stopword 블랙리스트 / token rarity scoring 은 후속 Phase. punctuation strip 으로 `결혼?` → `결혼`, `"경조금"` → `경조금` 매치 가능 |
| 매치 위치가 매우 끝 / 매우 앞이라 윈도우 한쪽이 잘림 | start/end 재조정 로직 (`start = max(0, end - length)`) 으로 윈도우 길이 보존. content 자체가 length 미만이면 그대로 반환 |
| 여러 키워드가 한 청크 안 멀리 떨어져 있을 때 한 윈도우로 못 다룸 | 첫 매치만 사용 — 보강 검색 필요시 LLM 이 query 다듬어 retry (7-2 프롬프트 가이드 그대로). 다중 윈도우 합성은 복잡도 vs 효용 trade-off 라 후속 Phase |
| Tool API 안 깨려고 dict wrapping 한 게 후속 도구에서 헷갈림 | tools_builtin.py 한 모듈 안에서만 보이므로 격리됨. dev log 에 "이건 retrieve 한정 우회 형태이고 Tool API 시그니처 자체는 안 바꿨다" 명시. 다른 도구도 query-aware summarize 가 필요해지는 시점에 Tool API 변경을 한 번에 |
| `_focus_window` 가 청크 본문에 `\n` / `\r` / `\t` 등 whitespace 가 섞여있을 때 매치 위치 계산 오차 | `_summarize_retrieve` 가 호출 전에 `content.replace('\n', ' ').strip()` 으로 정규화 — 7-2 그대로. window 안에 \r/\t 가 남으면 LLM 이 무시하므로 추가 처리 불필요 |

## 완료 정의 (Definition of Done)

- `_tokenize_query(query) -> list[str]` 가 punctuation strip + 길이≥2 + 길이 내림차순 정렬을 수행.
- `_focus_window(content, query, *, length)` 가 forward-bias 윈도우 단일 정책으로 동작:
  - 매치 < `length//4` → start=0 자연 클램프 → 첫 N자 (7-2 byte-identical)
  - `length//4 ≤ 매치 < length` → 매치 주변 윈도우 (값이 length 너머에 있어도 노출)
  - 매치 ≥ length → 매치 주변 윈도우 (본 목적)
  - 미매치 → 첫 N자 fallback
- `_retrieve_callable` 반환이 `{'query': ..., 'hits': [...]}` dict.
- `_summarize_retrieve` 가 query 를 받아 hit 마다 windowing 적용.
- `Tool.summarize` 시그니처 / 다른 도구 / Tool dataclass 변경 0.
- 단위 테스트 ≈17 신규 (FocusWindow ≈13 + retrieve 보강 ≈4), 전체 ≈327 green. 7-2 의 retrieve 테스트 한 건도 깨지지 않음.
- 채팅 UI smoke: 비교 질문에서 답 위치가 청크 401자+ 라도 LLM 이 답 가능 (캡처 1장).
- README §3-1 + §11 갱신 + Phase 7-3 dev log.
- Phase 7 마일스톤의 모든 이슈 (#43, #45, #47) 닫힘 → 마일스톤 다시 닫기 (이번엔 진짜 종료).
