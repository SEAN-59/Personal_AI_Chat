# 0.4.0 Phase 7 Agent 개발 설계

## 1. 목적

Phase 7의 목적은 `workflow`로 해결하기 어려운 질문을 처리할 수 있도록 `generic agent` 경로를 여는 것이다.

이 단계의 핵심 목표는 아래 4가지다.

1. `single_shot`과 `workflow`로 해결되지 않는 탐색형 질문을 처리한다.
2. ReAct 패턴 기반의 반복 실행 경로를 LangGraph 안에 도입한다.
3. agent가 직접 모든 것을 해결하는 구조가 아니라, 기존 `workflow`와 `service`를 도구처럼 재사용하도록 만든다.
4. 회사 전용 업무 로직이 아니라 범용 multi-hop 질문 처리 능력을 확보한다.

즉 이 단계는 `새로운 회사 업무 기능`을 넣는 단계가 아니라, `범용 agent 오케스트레이션`을 여는 단계다.

---

## 2. 현재 상태

Phase 6까지 끝난 시점의 가정 상태는 아래와 같다.

- `single_shot` 경로는 이미 LangGraph 안에서 안정적으로 동작한다.
- 질문 라우팅은 `single_shot / workflow / agent` 3분류를 지원한다.
- `workflow` 경로는 generic domain 기준으로 일부 실제 구현이 붙어 있다.
- `workflow`는 정해진 절차가 있는 문제를 처리할 수 있다.
- 아직 `agent` 경로는 실제로 열려 있지 않거나, 열려 있더라도 placeholder/fallback 수준이다.

즉 지금 남은 빈칸은 아래와 같다.

- 어떤 질문을 진짜 `agent`로 보내야 하는가
- agent가 어떤 도구를 어떻게 호출해야 하는가
- 반복 중 언제 종료해야 하는가
- agent가 `workflow`와 어떻게 역할을 나눌 것인가

따라서 Phase 7은 `generic agent runtime + tool orchestration`을 붙이는 단계다.

---

## 3. 이번 Phase의 방향

이번 Phase 7의 agent는 아래 성격을 가진다.

- 범용적이다
- 탐색형이다
- 도구 기반이다
- 반복 실행된다
- 근거 부족 시 답을 억지로 만들지 않는다

즉 아래 질문을 담당한다.

- 여러 자료를 비교해야 하는 질문
- 어떤 자료를 먼저 봐야 할지 고정되지 않은 질문
- 문서 탐색 → 재탐색 → 비교 → 정리 같은 multi-hop 질문
- 예외나 조건 분기가 많아 workflow 하나로 묶기 어려운 질문

반대로 아래 질문은 agent가 맡지 않는다.

- 문서 한 번 찾고 끝나는 질문 → `single_shot`
- 절차가 정해진 계산형 질문 → `workflow`
- 근거/입력이 부족해 답하면 안 되는 질문 → `unsupported`

즉 Phase 7의 agent는 `모든 어려운 질문의 만능 처리기`가 아니라, `정형화되지 않은 탐색형 질문의 상위 경로`여야 한다.

---

## 4. 이번 Phase의 범위

이번 Phase에서 하는 것:

- agent 경로의 책임 정의
- agent state 설계
- agent tool 인터페이스 설계
- ReAct loop 설계
- LangGraph 안에 `agent node` 연결
- agent 종료/실패/unsupported 정책 정의
- agent용 최소 응답 포맷 정의
- agent 테스트 설계

이번 Phase에서 하지 않는 것:

- 회사 전용 agent tool
- 사람 승인(HITL)
- 멀티 에이전트 협업
- 자가 계획 분해를 무제한으로 허용하는 자율 에이전트
- 장시간 백그라운드 실행
- 외부 SaaS observability 도입

즉 이번 Phase는 `안전한 1차 generic agent`를 만드는 단계다.

## 5. 핵심 설계 원칙

### 5-1. agent는 workflow 위에 올라가는 상위 레이어여야 한다

agent가 `workflow`를 대체하면 안 된다.

좋은 구조:

- `workflow` = 잘 정의된 작업 처리기
- `agent` = 어떤 처리기를 어떤 순서로 쓸지 결정하는 상위 오케스트레이터

즉 agent는 가능하면 직접 계산하지 않고, 이미 있는 `workflow`나 `core/service`를 tool처럼 호출해야 한다.

### 5-2. agent는 마지막 수단이어야 한다

라우팅 우선순위는 계속 아래를 유지한다.

1. `single_shot`
2. `workflow`
3. `agent`

즉 workflow로 풀 수 있는 질문을 agent로 보내면 안 된다.

### 5-3. unsupported를 agent로 숨기지 않는다

workflow가 없다고 해서 모든 계산형 질문을 agent로 보내면 안 된다.

예:

- 필요한 규칙이 없음
- 입력이 없음
- 문서 근거가 없음

이런 경우는 agent가 아니라 `unsupported` 또는 `추가 정보 필요`로 끝내야 한다.

### 5-4. 반복은 제한되어야 한다

agent loop는 반드시 제한을 둔다.

필수 제한:

- `max_iterations`
- 허용 tool 종류
- 각 step의 출력 형식
- 종료 조건

agent는 "생각하다가 끝없이 돌아가는 시스템"이 아니라, 제한된 범위 안에서만 도구를 반복 호출하는 구조여야 한다.

### 5-5. 도구 호출과 최종 응답은 분리한다

agent의 역할은 크게 두 개다.

- 다음에 무엇을 할지 결정
- 최종 답을 정리

하지만 retrieval, 계산, 표 조회 같은 실제 작업은 tool이 담당해야 한다.

즉 agent는 `판단자`, tool은 `실행자`다.

---

## 6. agent가 맡을 질문

권장 기준은 아래와 같다.

### 6-1. agent로 보내는 질문

- 여러 자료를 비교해야 한다
- 어떤 자료를 먼저 봐야 할지 유동적이다
- 예외/조건 분기가 많다
- 한 번의 retrieval이나 하나의 workflow로 끝나지 않는다

예:

- `이 두 문서 중 내 상황에 더 유리한 쪽을 비교해줘`
- `여러 자료를 종합해서 어떤 선택이 더 적절한지 알려줘`
- `표와 본문을 같이 봐서 규칙 차이를 설명해줘`

### 6-2. agent로 보내면 안 되는 질문

- 답이 문서에 거의 그대로 있는 질문
- 절차가 정해진 계산형 질문
- 근거가 부족해 답하면 안 되는 질문

예:

- `경조사 규정 알려줘` → single-shot
- `두 날짜 사이 며칠이야` → workflow
- `회사 규정도 없고 입력도 없는데 내 계산 결과를 알려줘` → unsupported

## 7. agent와 workflow의 관계

Phase 7에서는 아래 구조를 권장한다.

```text
single_shot
workflow
agent
```

그리고 agent 내부에서는 다음 tool들을 재사용한다.

- `retrieve_documents`
- `find_canonical_qa`
- `date_calculation workflow`
- `amount_calculation workflow`
- `table_lookup workflow`

즉 agent는 가능한 경우 아래처럼 행동해야 한다.

1. 먼저 문서를 찾아본다
2. 필요하면 이미 있는 workflow를 호출한다
3. 중간 결과를 모은다
4. 충분하면 최종 답을 만든다

이 구조를 지키면 agent가 기존 코드 위에 얹히는 형태가 된다.

---

## 8. Agent Tool 설계

초기 Phase 7에서 권장하는 tool 후보는 아래와 같다.

### 8-1. retrieval 계열

- `retrieve_documents(query)`
- `find_canonical_qa(query)`

용도:

- 자료 탐색
- 유사 질문 탐색

### 8-2. workflow wrapper 계열

- `run_workflow("date_calculation", input)`
- `run_workflow("amount_calculation", input)`
- `run_workflow("table_lookup", input)`

용도:

- agent가 직접 계산하지 않고 기존 workflow를 재사용

### 8-3. 경량 보조 tool

- `extract_relevant_tables(...)`
- `filter_sources(...)`
- `summarize_observations(...)`

이런 보조 tool은 필요할 때만 추가한다.

중요한 원칙은 이거다.

- tool은 작고 명확해야 한다
- tool은 가능한 deterministic해야 한다
- tool의 입력/출력은 구조화돼야 한다

---

## 9. Agent State 설계

권장 state 필드는 아래와 같다.

```python
{
  "question": "...",
  "history": [...],
  "route": "agent",
  "route_reason": "...",
  "agent_goal": "...",
  "plan": [],
  "observations": [],
  "tool_calls": [],
  "iteration_count": 0,
  "final_result": None,
  "error": None,
}
```

설명:

- `question`
  - 원본 사용자 질문
- `history`
  - 대화 맥락
- `agent_goal`
  - 이번 agent가 해결하려는 구체 목표
- `plan`
  - 현재까지 정리한 실행 단계
- `observations`
  - tool 결과 요약
- `tool_calls`
  - 어떤 tool을 어떤 인자로 불렀는지
- `iteration_count`
  - 반복 횟수
- `final_result`
  - 최종 결과
- `error`
  - 치명적 실패 메시지

이때 중요한 것은 state를 무한정 키우지 않는 것이다.

권장 원칙:

- tool 원문 전체를 계속 쌓지 않기
- 필요한 결과만 요약해서 유지
- 중간 state는 가능한 한 작게 유지

---

## 10. ReAct Loop 설계

Phase 7 agent의 기본 실행 흐름은 아래와 같다.

```text
question
-> agent state 초기화
-> think
-> choose tool
-> call tool
-> observe
-> decide done?
   -> yes: final answer
   -> no: next iteration
```

즉 아래 반복이다.

1. 현재 상태를 보고 다음 행동을 결정한다
2. tool을 하나 호출한다
3. 결과를 observation으로 저장한다
4. 충분한지 판단한다
5. 부족하면 다음 반복으로 간다

이 흐름이 곧 ReAct의 핵심이다.

---

## 11. 반복 제한과 종료 정책

Phase 7에서는 반드시 아래를 둔다.

### 11-1. `max_iterations`

권장 시작값:

- 3~5회

이유:

- 토큰 비용 제어
- 무한 루프 방지
- 응답 지연 방지

### 11-2. 종료 조건

아래 조건이면 종료한다.

- 최종 답을 만들 근거가 충분하다
- 더 유의미한 tool 호출이 없다
- 필요한 정보가 없어서 더 진행할 수 없다
- 반복 한도를 초과했다

### 11-3. 실패 조건

아래는 실패 또는 중단으로 본다.

- 연속 tool 실패
- 필수 자료 미확보
- 허용되지 않은 요청
- 결과 근거 부족

---

## 12. Agent 결과 상태

Phase 7에서도 결과 상태를 명확히 나누는 것이 좋다.

권장 상태:

- `OK`
- `NOT_FOUND`
- `UPSTREAM_ERROR`
- `UNSUPPORTED`

그리고 agent 특화 종료 사유도 내부적으로는 구분하는 편이 좋다.

예:

- `max_iterations_exceeded`
- `no_more_useful_tools`
- `insufficient_evidence`

이 값들은 최종적으로 사용자-facing 상태로 번역할 수 있다.

---

## 13. Agent 응답 원칙

agent 최종 응답은 아래 원칙을 따른다.

### 13-1. 답만이 아니라 근거 요약을 준다

가능하면 아래를 포함한다.

- 최종 판단
- 어떤 자료/도구를 봤는지
- 왜 그렇게 결론 내렸는지

### 13-2. 근거가 부족하면 부족하다고 말한다

agent가 그럴듯한 답을 지어내는 것을 막아야 한다.

즉:

- 확실하지 않으면 확실하지 않다고 말함
- 자료 부족이면 자료 부족으로 종료
- unsupported면 unsupported로 종료

### 13-3. 불필요하게 chain-of-thought를 노출하지 않는다

사용자에게는 내부 사고 전 과정을 장황하게 보여주기보다,

- 핵심 판단
- 근거 요약
- 필요한 다음 행동

정도로 정리하는 게 맞다.

## 14. LangGraph 연결 방향

Phase 7에서 graph는 최종적으로 아래 구조를 가지게 된다.

```text
START
-> router
  -> single_shot
  -> workflow
  -> agent
-> END
```

그리고 `agent` 노드는 내부적으로는 다시 반복 구조를 가진다.

즉 graph 바깥에서는 node 하나처럼 보이지만, 내부적으로는 ReAct loop가 돈다.

권장 구현 방향:

- `chat/graph/nodes/agent.py`
- `chat/agents/tools.py`
- `chat/agents/react.py`
- `chat/agents/result.py`

또는 services 안으로 묶고 싶다면:

- `chat/services/agent/`

중요한 건 `graph node`와 `agent runtime`을 분리하는 것이다.

- `graph node`
  - state 입출력 담당
- `agent runtime`
  - think / act / observe loop 담당

## 16. 구현 순서 제안

### Step 0. agent 경계 확정

- 어떤 질문을 agent로 보낼지 재정의
- workflow가 우선인 질문 정리
- unsupported로 막아야 하는 질문 정리

### Step 1. agent state와 결과 구조 정의

- state 필드 확정
- agent result/status 확정
- iteration 제한 정책 확정

### Step 2. tool 인터페이스 구축

- retrieval tool
- canonical QA tool
- workflow wrapper tool

### Step 3. ReAct runtime 구현

- think
- choose tool
- observe
- decide next
- stop

### Step 4. LangGraph agent node 연결

- router의 `agent` 경로를 실제 node로 연결
- 현재 fallback을 제거

### Step 5. reply/formatter 정리

- agent 결과를 사용자 응답 문자열로 변환
- 근거 요약 포함

### Step 6. 테스트

- multi-hop 질문
- 비교형 질문
- 예외형 질문
- tool 실패
- max iteration 초과
- unsupported 분기

---

## 17. 테스트 전략

최소한 아래를 포함해야 한다.

### 17-1. 정상 경로

- 문서 A/B 비교 질문
- retrieval 후 workflow 호출이 이어지는 질문
- table_lookup와 date/amount workflow를 혼합 사용하는 질문

### 17-2. 실패 경로

- tool 예외 발생
- 자료 없음
- parsing 실패
- iteration 초과

### 17-3. 안전 경로

- workflow로 가야 할 질문이 agent로 잘못 가지 않는지
- unsupported 질문에 agent가 억지 답을 만들지 않는지
- 근거 부족 시 정직하게 종료하는지

---

## 18. 완료 기준

Phase 7이 완료되었다고 보려면 최소한 아래가 충족돼야 한다.

- `agent` route가 실제로 열린다
- 제한된 ReAct loop가 동작한다
- 최소 2~3개의 tool을 실제로 사용할 수 있다
- 기존 workflow를 tool처럼 재사용한다
- `max_iterations`가 걸려 있다
- unsupported / not_found / upstream_error 정책이 있다
- 범용 multi-hop 질문 일부를 실제로 처리할 수 있다

---

## 19. 한 줄 정리

Phase 7은 `무엇이든 알아서 하는 자율 에이전트`를 만드는 단계가 아니라, `기존 single-shot과 workflow 위에 generic ReAct 오케스트레이션을 얹어 탐색형 질문까지 처리하는 단계`다.
