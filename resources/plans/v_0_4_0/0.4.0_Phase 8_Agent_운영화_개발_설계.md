# 0.4.0 Phase 8 Agent 운영화 개발 설계

## 1. 목적

Phase 8의 목적은 Phase 7에서 열린 `generic agent` 경로를 운영 가능한 수준으로 다듬는 것이다.

Phase 7까지의 핵심은 아래였다.

1. `single_shot / workflow / agent` 3분류 구조를 완성한다.
2. ReAct 기반 agent runtime을 만든다.
3. LangGraph 안에서 agent route가 실제로 동작하게 한다.
4. snippet windowing과 decision policy로 agent의 기본 안정성을 확보한다.

즉 Phase 7은 "agent가 동작한다"를 만드는 단계였다.

Phase 8은 여기서 한 단계 더 나아가 아래를 만든다.

1. agent 답변이 왜 그렇게 나왔는지 사용자와 운영자가 볼 수 있게 한다.
2. agent 실행 비용과 호출 목적을 추적 가능하게 한다.
3. 운영자가 BO에서 agent 동작 범위를 조정할 수 있게 한다.

즉 Phase 8은 `새로운 회사 전용 agent 기능`을 넣는 단계가 아니라, `generic agent 운영화` 단계다.

---

## 2. 현재 상태

Phase 7-4까지 끝난 시점의 기준 상태는 아래와 같다.

- `single_shot` 경로는 LangGraph 안에서 안정적으로 동작한다.
- `workflow` 경로는 generic workflow dispatch 기반으로 동작한다.
- `agent` 경로는 LangGraph의 `ROUTE_AGENT`에서 `agent_node`로 연결되어 동작한다.
- agent는 ReAct loop로 `retrieve_documents`, `find_canonical_qa`, `run_workflow` 도구를 사용할 수 있다.
- agent는 snippet windowing으로 긴 chunk의 답 위치를 더 잘 볼 수 있다.
- agent는 `[관련성 낮음]` 마커와 runtime guard로 무의미한 반복 retrieve를 줄인다.
- agent는 아직 사용자-facing 답변에서 sources와 tool trace를 충분히 드러내지 않는다.
- `TokenUsage`는 기록되지만 호출 목적 구분이 충분하지 않다.
- BO에서 agent 설정을 조정하거나 agent 도구 목록을 보는 기능은 아직 없다.

따라서 이제 남은 빈칸은 아래와 같다.

- agent 결과와 workflow 결과의 공통 구조를 어떻게 정리할 것인가
- agent가 사용한 근거와 도구 사용 내역을 어디까지 사용자에게 보여줄 것인가
- agent step 비용을 어떻게 분리 집계할 것인가
- 운영자가 agent를 켜고 끄거나 제한값을 조정할 수 있게 할 것인가

---

## 3. 이번 Phase의 방향

Phase 8은 agent의 사고 능력을 무작정 키우는 단계가 아니다.

이번 Phase의 방향은 아래다.

- 결과를 더 설명 가능하게 만든다.
- 비용을 더 추적 가능하게 만든다.
- 운영을 더 안전하게 만든다.
- BO에서 관리 가능한 범위를 넓힌다.
- 기존 `single_shot / workflow / agent` 경로의 회귀를 만들지 않는다.

즉 Phase 8은 `더 많은 자율성`보다 `더 높은 신뢰성과 운영성`을 우선한다.

## 5. Phase 8-1: Agent Result / Reply Surface 정리

### 5-1. 목적

Phase 8-1의 목적은 agent 답변의 결과 구조와 사용자-facing 표면을 정리하는 것이다.

현재 agent는 내부적으로 `WorkflowResult`를 재사용한다. 이는 Phase 7에서는 구현 비용을 줄이는 좋은 선택이었지만, 장기적으로는 아래 문제가 생긴다.

- workflow 결과와 agent 결과의 의미가 섞인다.
- agent가 어떤 도구를 사용했는지 reply layer에서 보기 어렵다.
- agent 답변에 sources를 붙이기 어렵다.
- 이후 agent 전용 상태나 trace를 넣을 위치가 애매하다.

따라서 8-1에서는 agent 결과 표면을 정리한다.

### 5-2. 주요 후보 작업

- `BaseResult` 추출 여부 검토
- `WorkflowResult`와 agent 결과의 공통 필드 정리
- agent result에 tool trace 또는 evidence 요약을 담을 위치 정의
- `build_reply_from_agent_result` 개선
- agent 답변에 sources를 어떻게 노출할지 정책 정의
- 사용자에게 보여줄 tool summary의 수준 정의

### 5-3. 중요한 설계 질문

- `AgentResult`를 새로 만들 것인가, 아니면 `WorkflowResult` 재사용을 더 유지할 것인가
- sources는 기존 `ChatLog.sources`와 같은 형식으로 맞출 것인가
- agent가 본 모든 chunk를 source로 보여줄 것인가, final_answer에 실제 사용한 것만 보여줄 것인가
- tool trace를 사용자에게 직접 보여줄 것인가, BO/로그 전용으로 둘 것인가

### 5-4. 권장 방향

초기 권장 방향은 아래다.

- `BaseResult`는 도입하되, 대규모 리팩터는 피한다.
- `WorkflowResult`의 public contract는 깨지 않는다.
- agent 전용 result를 만들더라도 reply layer에서만 점진적으로 사용한다.
- sources는 먼저 `retrieve_documents` 결과 기반으로 최소 노출한다.
- tool trace는 사용자에게 길게 노출하지 않고, 요약 또는 BO/로그용으로 남긴다.

즉 8-1의 목표는 "agent 답변을 화려하게 만들기"가 아니라, `결과 구조가 앞으로 확장 가능한 상태`를 만드는 것이다.

### 5-5. 제외

- 멀티 에이전트 trace UI
- LangSmith 같은 외부 trace UI
- 사용자가 agent chain-of-thought를 직접 보는 기능
- 회사 전용 tool 결과 포맷

## 6. Phase 8-2: TokenUsage Purpose / Agent Observability 기초

### 6-1. 목적

Phase 8-2의 목적은 OpenAI 호출 비용과 agent 실행 흐름을 더 잘 추적하는 것이다.

현재 `TokenUsage`는 토큰 사용량을 기록하지만, 호출 목적 구분이 약하다.

예를 들어 아래 호출들이 모두 비슷한 비용 로그로 보일 수 있다.

- single_shot 답변 생성
- query_rewriter
- workflow input extractor
- table_lookup LLM 셀 선택
- agent ReAct step
- agent final_answer 생성

운영 관점에서는 이것을 구분해야 한다.

### 6-2. 주요 후보 작업

- `TokenUsage`에 호출 목적 메타 필드 추가
- purpose 후보 enum 또는 문자열 정책 정의
- 기존 호출부에 purpose 전달
- BO 대시보드에서 purpose별 사용량을 볼 수 있게 할지 검토
- agent step별 token usage를 dev log 또는 내부 로그로 추적

### 6-3. 목적값 후보

초기 후보는 아래다.

```text
single_shot_answer
query_rewriter
workflow_extractor
workflow_table_lookup
agent_step
agent_final
prompt_management_test
unknown
```

세부 명칭은 detail 문서에서 실제 코드 구조를 보고 확정한다.

### 6-4. 중요한 설계 질문

- `purpose`를 DB enum처럼 강제할 것인가, 문자열로 열어둘 것인가
- 기존 TokenUsage row migration을 어떻게 처리할 것인가
- 과거 데이터의 purpose는 `unknown`으로 둘 것인가
- BO 대시보드에 바로 반영할 것인가, 먼저 DB와 서비스 layer만 반영할 것인가

### 6-5. 권장 방향

초기 권장 방향은 아래다.

- DB에는 문자열 필드로 추가한다.
- 선택지는 코드 상수로 관리한다.
- 기존 데이터는 `unknown` 또는 빈 값으로 안전 migration한다.
- Phase 8-2에서는 최소 집계까지만 하고, 복잡한 observability UI는 8-3 또는 후속 Phase로 넘긴다.

즉 8-2의 목표는 "멋진 모니터링 화면"이 아니라, `나중에 제대로 집계할 수 있는 데이터 기반`을 만드는 것이다.

### 6-6. 제외

- 외부 SaaS observability 연동
- LangSmith 도입
- Datadog / OpenTelemetry 연동
- agent step timeline UI
- 비용 예산 알림

## 7. Phase 8-3: BO Agent 운영 제어

### 7-1. 목적

Phase 8-3의 목적은 운영자가 BO에서 agent 동작을 안전하게 제어할 수 있게 하는 것이다.

현재 agent는 코드 상수와 라우팅 규칙에 의해 동작한다. 운영자가 BO에서 직접 조정할 수 있는 영역은 제한적이다.

Phase 8-3에서는 아래 같은 운영 제어를 검토한다.

- agent route 활성 여부
- `max_iterations`
- low relevance guard threshold
- 도구 카탈로그 확인
- agent용 router rule 관리 보강
- agent 실행 로그 또는 최근 실패 요약

### 7-2. 주요 후보 작업

- BO에 Agent 설정 페이지 추가
- agent global enable/disable 설정
- agent max_iterations 설정
- agent tool catalog 읽기 전용 표시
- agent 관련 RouterRule 관리 편의 보강
- 최근 agent 실패 원인 요약 표시

### 7-3. 중요한 설계 질문

- 설정을 DB에 둘 것인가, 환경변수에 둘 것인가
- 운영자가 잘못된 설정을 넣으면 어떻게 방어할 것인가
- 설정 변경이 즉시 반영되어야 하는가, 서버 재시작이 필요한가
- agent가 꺼져 있으면 `single_shot`으로 fallback할 것인가, unsupported로 막을 것인가

### 7-4. 권장 방향

초기 권장 방향은 아래다.

- Phase 8-3에서는 "읽기 쉬운 운영 제어"를 우선한다.
- 위험한 설정은 최소/최대 범위를 둔다.
- agent disable 시에는 기본적으로 `single_shot` fallback을 우선 검토한다.
- tool catalog는 먼저 읽기 전용으로 공개한다.
- 쓰기 가능한 tool 제어는 후속 Phase로 미룬다.

즉 8-3의 목표는 "운영자가 agent를 안전하게 관리할 수 있는 첫 BO 표면"을 만드는 것이다.

### 7-5. 제외

- tool별 세부 권한 관리
- 사용자별 agent 활성 정책
- agent 실행 step 전체 timeline UI
- 실시간 streaming 상태 표시
- 외부 observability SaaS 임베딩

## 8. Phase 8 전체 설계 원칙

### 8-1. 회귀 0 우선

Phase 8은 이미 동작하는 `single_shot / workflow / agent` 경로를 망가뜨리면 안 된다.

특히 아래는 유지해야 한다.

- single_shot 답변 포맷
- workflow reply 포맷
- agent route 진입 조건
- RouterRule 동작
- Prompt 관리 BO
- TokenUsage 기본 집계

### 8-2. agent chain-of-thought는 노출하지 않는다

Phase 8에서 "agent trace"를 다루더라도 내부 reasoning 전체를 사용자에게 노출하지 않는다.

노출 가능한 것은 아래 수준이다.

- 어떤 도구를 사용했는지
- 어떤 문서 근거를 참조했는지
- 충분한 근거가 없어서 답하지 못했다는 사실

노출하지 않는 것:

- 모델 내부 reasoning 전문
- ReAct reasoning 원문
- prompt 전체

### 8-3. 운영자용 정보와 사용자용 정보를 분리한다

사용자에게 보여줄 정보와 BO에서 운영자가 볼 정보는 다르다.

사용자용:

- 짧은 출처
- 답변 근거 요약
- 자료 부족 안내

운영자용:

- tool call 목록
- failure_kind
- token purpose
- 라우팅 결정
- 설정값

### 8-4. 설정은 안전한 기본값을 가져야 한다

BO 설정을 도입하더라도 기본값은 코드 상수와 동일하거나 더 안전해야 한다.

예:

- agent가 비활성화되어도 서비스 전체가 실패하지 않는다.
- max_iterations는 최소/최대 범위를 가진다.
- 잘못된 설정은 fallback 또는 validation error로 막는다.

## 9. 상세 플랜 작성 원칙

Phase 8의 각 detail 문서는 아래 기준을 따라야 한다.

### 9-1. 각 detail 문서는 독립적으로 리뷰 가능해야 한다

8-1, 8-2, 8-3은 서로 연결되지만, 각 문서는 독립 PR처럼 검토 가능해야 한다.

각 detail 문서에는 최소한 아래가 있어야 한다.

- 현재 코드 기준 상태
- 변경 범위
- 제외 범위
- 파일 변경 예상
- migration 여부
- 테스트 계획
- 사용자-facing smoke
- 회귀 위험
- Definition of Done

### 9-2. 구현 전에 설계 review를 먼저 통과한다

Phase 8부터는 agent 운영 표면을 건드리므로, detail 문서가 먼저 review 되어야 한다.

권장 흐름:

1. Claude가 detail 문서 초안 작성
2. Codex가 문서 리뷰
3. finding 반영
4. 개발 시작
5. 개발 후 코드 리뷰

즉 이 상위 문서는 작업 순서를 고정하는 게 아니라, review 가능한 개발 단위를 정의한다.

### 9-3. 테스트 수는 detail 문서에서 확정한다

이 문서에서는 테스트 개수를 고정하지 않는다.

이유:

- 8-1은 구조 리팩터 성격이라 테스트 범위가 실제 코드 의존적이다.
- 8-2는 migration 여부에 따라 테스트 수가 달라진다.
- 8-3은 BO view/form/template 범위에 따라 테스트 수가 달라진다.

따라서 테스트 수는 각 detail 문서에서 실제 파일을 확인한 뒤 확정한다.

## 10. Phase 8 완료 정의

Phase 8 전체 완료 기준은 아래다.

- agent 답변 결과 구조가 workflow와 무리 없이 공존한다.

- BO에서 agent 운영 설정 또는 상태를 최소한 확인할 수 있다.
- 기존 `single_shot / workflow / agent` 경로의 주요 회귀 테스트가 통과한다.
- README와 개발 로그에 Phase 8 변경점이 기록된다.

## 12. 요약

Phase 8의 핵심은 아래 한 줄로 정리할 수 있다.

> Phase 7에서 만든 generic agent를, 사용자가 믿고 운영자가 관리할 수 있는 형태로 다듬는다.

Phase 8-1은 결과와 답변 표면을 정리한다.

Phase 8-2는 비용과 실행 목적을 추적 가능하게 만든다.

Phase 8-3은 BO에서 agent를 안전하게 운영할 수 있는 표면을 만든다.
