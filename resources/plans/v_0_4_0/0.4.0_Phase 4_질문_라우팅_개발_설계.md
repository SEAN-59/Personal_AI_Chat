# 0.4.0 Phase 4 질문 라우팅 개발 설계

## 1. 목적

Phase 4의 목적은 사용자 질문을 `single-shot / workflow / agent` 중 어떤 처리 경로로 보낼지 판단하는 라우팅 구조를 만들고, 운영자가 BO에서 라우팅 키워드와 규칙을 관리할 수 있는 기반을 마련하는 것이다.

Phase 4는 두 단계로 나눈다.

1. `Phase 4-1. 질문 라우팅 Core`
   - route 정의
   - 규칙 기반 router 함수
   - LangGraph router node 연결
   - conditional edge/fallback 정책

2. `Phase 4-2. BO Router Rule 관리`
   - workflow/agent 키워드 관리
   - rule 활성/비활성
   - 우선순위 관리
   - BO 관리 화면
   - router가 동적 rule을 읽도록 확장

이렇게 나누는 이유는 먼저 라우팅 구조를 안정화하고, 그 다음 운영자가 조정 가능한 관리 UI를 붙이는 것이 더 안전하기 때문이다.

---

## 2. 현재 상태

Phase 2~3 이후의 가정 상태는 아래와 같다.

- 질문 실행은 LangGraph를 통해 시작된다.
- graph에는 router node가 존재한다.
- Phase 2의 router는 placeholder로, 항상 `single_shot`만 선택한다.
- Phase 3을 통해 single-shot 내부 구조는 어느 정도 정리되었다.

Phase 4에서는 이 placeholder router를 실제 의도 분류기로 확장한다.

단, 이 단계에서는 workflow와 agent의 모든 구현을 완료하지 않는다. 대신 어떤 질문이 어떤 경로로 가야 하는지 명확히 정의하고, 최소한 fallback 가능한 라우팅 구조를 만든다.

---

## 3. 최종 3분류 정의

### 3-1. single-shot

정의:

- 업로드된 문서와 CanonicalQA 검색 결과만으로 바로 답변 가능한 질문
- 검색 1회와 LLM 1회 호출로 충분한 질문

예시:

- `연차 규정이 뭐야?`
- `경조사 휴가는 며칠이야?`
- `복지포인트 사용 기준 알려줘`
- `개인정보 처리방침에 보관 기간이 어떻게 되어 있어?`

특징:

- 답이 문서에 직접 있거나, 몇 개 청크를 종합하면 됨
- 개인 데이터 조회가 필요 없음
- 계산 절차가 복잡하지 않음

### 3-2. workflow

정의:

- 처리 순서와 필요한 정보가 비교적 정해져 있는 질문
- 공식, 규칙, 날짜/금액/기간 계산이 필요한 질문
- 정형 입력값을 모아 코드로 계산하는 것이 더 안정적인 질문

예시:

- `퇴직금 얼마야?`
- `입사일 기준으로 근속연수 계산해줘`
- `올해 내 연차 며칠이야?`
- `최근 3개월 평균 급여 기준으로 계산해줘`

특징:

- 여러 단계가 필요하지만 순서가 대체로 고정됨
- 일반 ReAct agent보다 정형 workflow가 더 정확하고 저렴함
- 필요한 값이 없으면 사용자에게 추가 입력을 요청하거나 지원 불가 응답을 해야 함

### 3-3. agent

정의:

- 처리 경로가 미리 고정되어 있지 않고, 검색/조회/비교/재검색이 유동적인 질문
- 무엇을 먼저 찾아야 할지 모델의 판단이 필요한 질문
- 여러 문서, 조건, 예외를 비교하며 탐색해야 하는 질문

예시:

- `내 상황이면 육아휴직과 단축근무 중 뭐가 더 유리해?`
- `이 규정들이 서로 충돌하는 것 같은데 어떻게 해석해야 해?`
- `내 조건에 맞는 제도를 비교해서 추천해줘`
- `A 케이스와 B 케이스를 규정 기준으로 비교해줘`

특징:

- 질문 해석과 다음 행동 선택이 유동적
- tool 호출이 반복될 수 있음
- 비용과 불확실성이 크므로 마지막 경로로 둔다

## 4. Phase 4-1 질문 라우팅 Core

### 4-1-1. 범위

Phase 4-1에서는 아래를 한다.

- route 값 정의
- 규칙 기반 router 함수 작성
- LangGraph router node에 연결
- route 결과를 state에 기록
- workflow/agent 미구현 상태의 fallback 정책 정의
- 대표 질문셋으로 라우팅 검증

Phase 4-1에서 하지 않는 것:

- BO에서 rule 수정
- DB 기반 rule 모델
- 고도화된 LLM router
- workflow/agent 본체 구현

즉 4-1은 `일단 코드 기반 라우터로 3분류 구조를 동작시키는 단계`다.

### 4-1-2. route 값

권장 route 값:

- `single_shot`
- `workflow`
- `agent`

추후 확장 후보:

- `clarify`
- `unsupported`

Phase 4-1에서는 우선 3개 route만 사용한다.

### 4-1-3. 라우팅 우선순위

라우팅은 아래 순서로 판단한다.

1. 명확한 workflow intent가 있으면 `workflow`
2. 명확한 agent intent가 있으면 `agent`
3. 나머지는 기본적으로 `single_shot`

이 순서가 중요한 이유:

- workflow는 정형 처리라 안정성이 높음
- agent는 가장 비싸고 불확실하므로 신중히 사용
- single-shot은 현재 안정적으로 동작하는 기본 경로

즉, 라우터의 기본 철학은 아래와 같다.

```text
명확한 정형 업무 -> workflow
명확한 탐색/비교/추천 -> agent
그 외 -> single-shot
```

### 4-1-4. workflow 신호

아래 표현이 있으면 workflow 가능성이 높다.

- `계산`
- `산정`
- `얼마`
- `몇 일`
- `몇 년`
- `평균`
- `합계`
- `차감`
- `근속`
- `퇴직금`
- `연차 계산`
- `잔여 연차`
- `급여`
- `수당`
- `입사일`
- `퇴사일`
- `최근 3개월`

workflow로 보내는 조건:

- 위 키워드가 있고
- 정해진 업무 템플릿에 매핑 가능할 때

주의:

- 단순히 `얼마`가 있다고 무조건 workflow로 보내면 안 된다.
- 예: `복지포인트는 얼마야?`는 문서에 정해진 금액이 있으면 single-shot일 수 있다.

### 4-1-5. agent 신호

아래 표현이 있으면 agent 가능성이 높다.

- `비교`
- `추천`
- `유리`
- `불리`
- `종합`
- `예외`
- `케이스`
- `해석`
- `충돌`
- `만약`
- `내 상황`
- `어떤 게 나아`

agent로 보내는 조건:

- 단순 문서 응답이나 정형 workflow로 처리하기 어렵고
- 여러 자료를 비교/탐색해야 하며
- 다음 행동이 고정되어 있지 않을 때

주의:

- `비교`가 있어도 두 항목 설명 정도면 single-shot으로 가능할 수 있다.
- agent는 마지막 수단으로 둔다.

### 4-1-6. single-shot 기본 조건

아래는 single-shot으로 보내는 것이 기본이다.

- 정의 질문
- 정책/규정 설명 질문
- 절차 안내 질문
- 기준 확인 질문
- 문서 내 수치 확인 질문

예:

- `연차 기준이 뭐야?`
- `경조사 휴가 종류 알려줘`
- `결재 라인은 어떻게 돼?`
- `개인정보 보관 기간은?`

### 4-1-7. 코드 기반 router 구현

초기 구현은 규칙 기반으로 한다.

개념:

```python
def route_question(question: str) -> RouteDecision:
    if matches_workflow(question):
        return RouteDecision(route="workflow", reason="workflow_keyword")
    if matches_agent(question):
        return RouteDecision(route="agent", reason="agent_keyword")
    return RouteDecision(route="single_shot", reason="default")
```

`RouteDecision`에 담을 후보 필드:

- `route`
- `reason`
- `matched_rules`
- `confidence`

Phase 4-1에서는 `route`와 `reason`만 있어도 충분하다.

### 4-1-8. LangGraph 연결

Phase 4-1 이후 graph 구조는 아래 방향으로 확장된다.

```text
START
  -> router
  -> single_shot
  -> END

router
  -> workflow
  -> END

router
  -> agent
  -> END
```

다만 workflow/agent node가 아직 완성되지 않았다면 아래 방식 중 하나를 선택한다.

1. route만 기록하고 아직 `single_shot`으로 fallback
2. placeholder node에서 `아직 지원 준비 중` 안내 응답 반환

권장:

- Phase 4-1에서는 route 판단까지 구현
- 실제 workflow/agent 실행은 이후 Phase에서 활성화
- 사용자 경험을 깨지 않기 위해 미구현 route는 우선 `single_shot fallback`을 허용

### 4-1-9. Phase 4-1 코드 적용 대상

신규 생성 후보:

- `chat/services/question_router.py`
- `chat/graph/routes.py`

수정 대상:

- `chat/graph/state.py`
  - `route`, `route_reason`, `matched_rules` 후보 필드 정리

- `chat/graph/nodes/router.py`
  - placeholder에서 실제 router 함수 호출로 변경

- `chat/graph/app.py`
  - conditional edge 또는 fallback edge 설계

### 4-1-10. Phase 4-1 세부 개발 순서

Step 1. route 상수 정의

- `single_shot`, `workflow`, `agent` 중앙화

Step 2. 규칙 기반 router 함수 작성

- workflow keyword 규칙
- agent keyword 규칙
- 기본값 single-shot

Step 3. router node 연결

- graph router node에서 route 함수 호출
- state에 route/reason 기록

Step 4. fallback 정책 적용

- workflow/agent 미구현 상태에서 single-shot fallback 또는 placeholder 결정

Step 5. 예시 질문셋 검증

- single-shot/workflow/agent/애매한 질문 예시로 기대 route 확인

## 5. Phase 4-2 BO Router Rule 관리

### 5-1. 목적

Phase 4-2의 목적은 Phase 4-1에서 코드 기반으로 잡은 라우팅 신호를 BO에서 관리할 수 있게 만드는 것이다.

즉, 운영자가 아래를 직접 관리할 수 있게 한다.

- workflow 키워드
- agent 키워드
- rule 활성/비활성
- 우선순위
- 설명/메모

이 단계가 끝나면 라우터는 코드에 박힌 키워드만 보는 것이 아니라, BO에서 관리되는 rule을 읽어 route를 판단할 수 있다.

### 5-2. 왜 BO 관리가 필요한가

운영 중에는 새로운 질문 패턴이 계속 생긴다.

예:

- `퇴직 정산`
- `잔여휴가`
- `평균임금`
- `어떤 제도가 더 나아`

이런 표현을 매번 코드 수정으로 추가하면 배포가 필요하다. BO에서 rule을 관리하면 운영자가 질문 패턴을 더 빠르게 반영할 수 있다.

### 5-3. 저장 방식 선택

BO에서 rule을 관리하려면 DB 기반이 가장 자연스럽다.

파일 기반도 가능하지만 아래 이유로 DB 기반을 권장한다.

- 활성/비활성 관리가 쉬움
- 우선순위 정렬이 쉬움
- BO 폼과 잘 맞음
- 향후 수정 이력/작성자/검증 상태 확장이 쉬움

Phase 4-2 권장 방향:

- `RouterRule` 모델 추가
- BO에서 CRUD 관리
- router가 enabled rule을 읽어 판단

### 5-4. RouterRule 모델 설계

권장 필드:

- `name`
  - rule 이름
- `route`
  - `single_shot`, `workflow`, `agent`
- `match_type`
  - `contains`, `regex`, `exact`
- `pattern`
  - 매칭할 키워드/패턴
- `priority`
  - 숫자가 높을수록 우선
- `enabled`
  - 활성 여부
- `description`
  - 운영 메모
- `created_at`
- `updated_at`

초기에는 `contains`만 지원해도 된다.

확장 후보:

- `regex`
- `negative pattern`
- `intent group`
- `confidence weight`
- `created_by`
- `updated_by`

### 5-5. rule 평가 정책

권장 평가 순서:

1. enabled rule만 조회
2. priority 높은 순으로 정렬
3. pattern match
4. 가장 먼저 매칭된 rule의 route 선택
5. 아무 rule도 없으면 `single_shot`

주의:

- workflow와 agent rule이 동시에 걸릴 수 있으므로 우선순위가 필요하다.
- `single_shot` rule은 처음에는 만들지 않아도 된다. 기본값이 single-shot이기 때문이다.

### 5-6. BO 화면 설계

예상 메뉴 위치:

- BO 사이드바에 `Router Rule 관리` 또는 `라우팅 관리` 추가

목록 화면:

- rule 이름
- route
- match_type
- pattern
- priority
- enabled
- updated_at
- 수정 버튼

생성/수정 화면:

- name
- route select
- match_type select
- pattern input
- priority input
- enabled checkbox
- description textarea

권장 기능:

- 활성/비활성 빠른 토글
- priority 기준 정렬
- route별 필터

Phase 4-2에서는 복잡한 테스트 UI까지는 제외한다.

추후 후보:

- 질문을 입력하면 어떤 rule이 매칭되는지 미리보기
- rule 충돌 감지
- 변경 이력

### 5-7. RouterRule 관리 URL 설계

예상 URL:

- `/bo/router-rules/`
- `/bo/router-rules/new/`
- `/bo/router-rules/<int:pk>/edit/`
- `/bo/router-rules/<int:pk>/toggle/`
- `/bo/router-rules/<int:pk>/delete/`

현재 BO 구조에 맞춰 function view 기반으로 시작해도 충분하다.

### 5-8. router와 DB rule 연결

Phase 4-2 이후 router는 아래 순서로 판단한다.

1. DB RouterRule 조회
2. 매칭 rule 있으면 해당 route 반환
3. 없으면 Phase 4-1의 코드 기본 규칙 사용
4. 그래도 없으면 `single_shot`

이렇게 하면 DB rule이 운영 설정 역할을 하고, 코드 규칙은 fallback 역할을 한다.

### 5-9. 캐싱 정책

RouterRule은 매 요청 DB 조회해도 초기에는 큰 문제 없을 수 있다.

다만 운영 안정성을 위해 아래 중 하나를 선택한다.

옵션 1. 매 요청 DB 조회

- 구현 단순
- BO 수정 즉시 반영
- rule 수가 적을 때 적합

옵션 2. 짧은 TTL 캐시

- DB 부하 감소
- 수정 반영이 약간 지연될 수 있음

Phase 4-2 권장:

- 초기에는 매 요청 DB 조회
- rule 수가 많아지면 TTL 캐시 검토

### 5-10. Phase 4-2 코드 적용 대상

신규 생성 후보:

- `chat/models.py`
  - `RouterRule` 모델 추가

- `chat/migrations/...`
  - RouterRule migration

- `bo/views/router_rules.py`
- `bo/templates/bo/router_rules.html`
- `bo/templates/bo/router_rule_form.html`

수정 대상:

- `bo/urls.py`
- `bo/views/__init__.py`
- `bo/templates/bo/base.html`
- `chat/services/question_router.py`

### 5-11. Phase 4-2 세부 개발 순서

Step 1. RouterRule 모델 추가

- route/match_type choices 정의
- priority/enabled 필드 추가

Step 2. migration 생성

- DB 스키마 반영

Step 3. router rule 조회 로직 작성

- enabled rule 조회
- priority 정렬
- contains match

Step 4. BO 목록/폼 구현

- 목록
- 생성
- 수정
- 삭제 또는 비활성화

Step 5. router와 DB rule 연결

- DB rule 우선
- 코드 rule fallback
- default single-shot

Step 6. 검증

- BO에서 workflow 키워드 추가
- 해당 질문이 workflow route로 분류되는지 확인
- rule 비활성화 후 route 변경 확인

## 6. 질문 예시 분류표

| 질문 | 기대 route | 이유 |
| --- | --- | --- |
| 연차 규정이 뭐야? | single_shot | 문서 설명형 |
| 경조사 휴가는 며칠이야? | single_shot | 문서 내 수치 확인 |
| 복지포인트 사용 기준 알려줘 | single_shot | 정책 설명형 |
| 퇴직금 얼마야? | workflow | 계산/정형 업무 |
| 입사일 기준으로 근속연수 계산해줘 | workflow | 날짜/기간 계산 |
| 최근 3개월 급여 평균 내줘 | workflow | 평균 계산 |
| 올해 내 연차 며칠이야? | workflow | 개인 조건 + 산정 |
| 육아휴직이랑 단축근무 중 뭐가 더 유리해? | agent | 비교/추천/상황 판단 |
| 규정 A와 B가 충돌하는 것 같은데 어떻게 해석해? | agent | 예외/충돌 해석 |
| 내 상황에 맞는 제도 추천해줘 | agent | 경로가 유동적인 탐색형 |

## 7. 애매한 질문 처리 원칙

### 7-1. 기본은 single-shot

애매하면 우선 single-shot으로 보낸다.

이유:

- 현재 가장 안정적인 경로
- 비용이 낮음
- 문서 기반 답변으로도 해결될 가능성이 있음

### 7-2. 계산형인데 필요한 값이 없으면 workflow에서 중단 가능

예:

- `퇴직금 얼마야?`

이 질문은 workflow로 갈 수 있지만, 실제 계산에 필요한 값이 없으면 아래 중 하나를 한다.

- 필요한 입력값을 사용자에게 요청
- 현재 자료만으로는 계산 불가하다고 답변

### 7-3. agent는 명확한 신호가 있을 때만 사용

agent는 비용과 복잡도가 크므로 아래 조건이 강할 때만 사용한다.

- 비교/추천/충돌/예외 해석
- 여러 tool이 필요함
- 다음 조회가 유동적임

## 8. 실패 시 승급 정책

처음 route가 항상 정답일 수는 없다. 따라서 실패 신호에 따라 상위 경로로 승급할 수 있다.

### 8-1. single-shot -> workflow 승급

승급 조건:

- 질문에 계산/산정/기간/금액 신호가 있음
- single-shot 답변이 구체적인 값을 내지 못함
- 답변이 설명만 하고 계산 결과가 없음

### 8-2. workflow -> agent 승급

승급 조건:

- 정해진 workflow로 처리하려 했지만 규정 충돌/예외 해석이 필요함
- 필요한 절차가 고정되어 있지 않음
- 여러 제도 비교/추천이 필요함

### 8-3. 승급은 무제한 반복하지 않는다

권장 원칙:

- 한 요청에서 route 승급은 최대 1회
- agent까지 갔으면 더 이상 승급하지 않음
- 무한 루프 방지를 위해 명시적 제한 필요

## 9. 예상 위험 요소

### 9-1. 키워드 기반 오분류

위험:

- `얼마` 같은 단어만 보고 workflow로 잘못 보낼 수 있음

대응:

- 키워드 단독 조건보다 업무 intent 조합으로 판단
- 오분류 사례를 지속적으로 누적

### 9-2. BO rule 오설정

위험:

- 운영자가 잘못된 pattern을 등록하면 route가 의도치 않게 바뀔 수 있음

대응:

- enabled 토글 제공
- priority 명확화
- rule description 작성 권장
- 추후 미리보기/충돌 감지 기능 고려

### 9-3. agent 과사용

위험:

- 비교/추천 단어가 있다고 너무 쉽게 agent로 보내면 비용이 증가한다.

대응:

- agent는 명확한 탐색형 질문에만 사용
- 단순 비교 설명은 single-shot 유지

### 9-4. 아직 구현되지 않은 route로 분기

위험:

- workflow/agent node가 준비되지 않았는데 route만 먼저 분기하면 사용자 응답이 깨질 수 있음

대응:

- Phase 4-1에서는 fallback 정책을 반드시 둔다.
- Phase 5~7에서 실제 node가 준비되면 fallback을 제거한다.

## 10. 완료 정의

Phase 4는 아래 조건을 만족하면 완료로 본다.

Phase 4-1 완료 조건:

- `single_shot / workflow / agent` route 기준이 정의되었다.
- 규칙 기반 router 함수가 구현 가능하게 설계되었다.
- LangGraph router node가 실제 route 값을 state에 기록할 수 있다.
- 대표 질문셋에 대한 route 기대값이 정리되었다.
- workflow/agent 미구현 상태의 fallback 정책이 정해졌다.

Phase 4-2 완료 조건:

- RouterRule 모델 또는 이에 준하는 운영 관리 구조가 설계되었다.
- BO에서 router rule을 생성/수정/비활성화할 수 있는 화면이 정의되었다.
- router가 BO rule을 읽어 route 판단에 사용할 수 있는 구조가 정의되었다.
- rule 오설정에 대한 최소 안전장치가 정리되었다.

## 11. Phase 4 완료 후 다음 단계 연결

Phase 4가 끝나면 다음 단계인 `Workflow Core 개발`에서 아래 이점을 얻는다.

- 어떤 질문이 workflow로 들어올지 기준이 생김
- workflow가 요구할 공통 계산 기능을 더 명확히 정의 가능
- agent로 보낼 질문과 workflow로 처리할 질문의 경계가 생김
- BO를 통해 라우팅 신호를 운영 중 조정할 수 있음

즉 Phase 4는 `3분류 구조의 문`을 만드는 단계이며, Phase 5 이후는 각 문 뒤의 실제 처리 로직을 채우는 단계로 본다.
