# 2.0.0 Phase 6 범용 Workflow Domain 개발 설계

## 1. 목적

Phase 6의 목적은 Phase 5에서 만든 `Workflow Core` 위에 실제로 동작하는 `workflow domain`을 올리는 것이다.

다만 이번 Phase 6의 방향은 회사 전용 업무 로직을 붙이는 것이 아니다.

이번 단계의 핵심 목표는 아래 4가지다.

1. `workflow`를 회사 전용 계산기가 아니라 `범용 문제 해결 경로`로 정의한다.
2. 실제 workflow가 식별될 수 있도록 `workflow_key` 체계를 도입한다.
3. `workflow / agent / unsupported`의 경계를 명확히 한다.
4. Phase 7의 agent가 오기 전까지도 유용한 범용 multi-step 처리를 제공한다.

즉 이 단계는 `회사 규칙 workflow`가 아니라 `범용 질문 유형 workflow`를 만드는 단계다.

---

## 2. 현재 상태

현재 기준 상태는 아래와 같다.

- LangGraph는 이미 적용되어 있다.
- 질문은 `single_shot / workflow / agent`로 분류될 수 있다.
- Phase 4의 라우터는 실제로 동작하지만, 현재는 `workflow`와 `agent` 모두 내부적으로 `single_shot`으로 포워딩된다.
- Phase 5를 통해 `chat/workflows/core/`가 구축되어 있다.
- 아직 `chat/workflows/domains/`에는 실제 domain workflow 구현이 없다.

즉 지금은

- `분류는 된다`
- `공통 엔진도 있다`
- `실제 workflow domain만 비어 있다`

는 상태다.

따라서 이제 필요한 것은 새 Phase를 더 쪼개는 것이 아니라, `generic domain workflow`를 정의하고 붙이는 것이다.

---

## 3. 이번 Phase의 방향

이번 Phase 6에서는 기존에 논의했던 `퇴직금`, `연차`, `수당` 같은 회사 전용 workflow를 넣지 않는다.

대신 아래처럼 범용 질문 유형 중심의 workflow를 만든다.

- 날짜 계산형
- 금액 계산형
- 표 조회형
- 문서 비교형
- 조건 분기형
- 다중 자료 종합형

즉 이번 Phase의 `domain`은

- 회사 부서/정책 도메인

이 아니라

- 질문 해결 패턴 도메인

을 의미한다.

---

## 4. 이번 Phase의 범위

이번 Phase에서 하는 것:

- `workflow_key` 개념 정의
- generic workflow 후보군 확정
- `route=workflow`일 때 어떤 domain workflow로 들어갈지 연결 구조 설계
- generic workflow 1차 구현
- `agent`로 보낼 질문과 `unsupported`로 막을 질문 기준 정리
- Phase 4 라우터와 Phase 6 workflow의 연결 지점 설계

이번 Phase에서 하지 않는 것:

- 퇴직금 공식 구현
- 연차 산정 규칙 구현
- 특정 회사 규정 계산
- 외부 인사/급여 시스템 연동
- ReAct agent 본체 구현
- 사람 승인(human-in-the-loop)

즉 이번 Phase는 `범용 workflow 실제 구현 단계`이지, `회사 업무 자동화 단계`가 아니다.

---

## 5. 핵심 설계 원칙

### 5-1. domain은 문제 유형이어야 한다

이번 Phase에서의 domain은 아래처럼 정의한다.

- `date_calculation`
- `amount_calculation`
- `table_lookup`
- `document_compare`
- `conditional_reasoning`
- `multi_source_summary`

반대로 아래처럼 특정 조직 정책을 직접 뜻하는 이름은 이번 Phase에서 다루지 않는다.

- `retirement_pay`
- `annual_leave`
- `allowance_policy`
- `company_hr_rules`

### 5-2. workflow는 정해진 절차가 있는 질문만 맡는다

workflow는 다음에 무엇을 해야 할지가 비교적 명확한 질문을 맡는다.

예:

- 날짜 두 개를 받아 기간 계산
- 금액 목록을 받아 합계/평균 계산
- 표에서 특정 행/열 값 조회
- 문서 두 개의 특정 항목 비교

반대로 절차가 유동적이거나 탐색이 필요한 질문은 Phase 7 agent로 넘긴다.

### 5-3. agent를 workflow의 쓰레기통으로 쓰지 않는다

workflow가 없다고 해서 모든 계산형 질문을 agent로 보내면 안 된다.

예:

- 규칙이 전혀 없는 계산 요청
- 필요한 입력이 없는 계산 요청
- 문서 근거 없이 추정해야 하는 요청

이런 것은 `agent`보다 `unsupported` 또는 `추가 정보 필요`로 처리해야 한다.

### 5-4. Phase 6의 기준 정리는 Step 0에서 먼저 한다

Phase 6에 들어가면 가장 먼저 아래를 정리해야 한다.

- 회사 전용 예시/키워드 제거
- generic workflow 목록 확정
- `workflow_key` 체계 정의
- `agent`와 `unsupported` 경계 정의

즉 구현보다 먼저 기준부터 정리한다.

## 6. 라우팅 구조 재정의

Phase 4까지는 사실상 `route`만 있었다.

```python
route = "single_shot" | "workflow" | "agent"
```

하지만 Phase 6부터는 이것만으로 부족하다.

이제는 `workflow`로 들어간 뒤 어떤 workflow를 타는지도 알아야 한다.

따라서 추천 구조는 아래와 같다.

```python
{
  "route": "workflow",
  "workflow_key": "date_calculation",
  "route_reason": "...",
  "matched_rules": [...],
}
```

즉 앞으로는

- `route`
  - 큰 처리 경로
- `workflow_key`
  - 어떤 workflow domain을 탈지
- `route_reason`
  - 왜 그렇게 분류됐는지
- `matched_rules`
  - 어떤 rule/keyword가 작동했는지

를 함께 본다.

---

## 7. unsupported 정책

Phase 6부터는 `workflow / agent / unsupported` 경계가 필요하다.

권장 기준은 아래와 같다.

### 7-1. workflow로 보내는 질문

- 정해진 generic workflow가 존재한다
- 필요한 입력이 충분하거나, 누락 여부를 명확히 판단할 수 있다
- 절차가 고정적이다

예:

- `2024-01-01부터 2024-03-31까지 며칠이야?`
- `이 세 금액 합계가 얼마야?`
- `표에서 A항목 금액 찾아줘`

### 7-2. agent로 보내는 질문

- 어떤 자료를 먼저 봐야 할지 유동적이다
- 탐색, 비교, 종합, 예외 검토가 필요하다
- workflow 하나로 고정하기 어렵다

예:

- `이 두 문서 중 내 상황에 더 유리한 쪽을 추천해줘`
- `여러 규정을 종합해서 어떤 선택이 좋은지 비교해줘`

### 7-3. unsupported로 보내는 질문

- 정확한 규칙이나 근거가 없다
- 필요한 입력이 없고 추정하면 안 된다
- generic workflow도 아니고 agent로 가도 정답 근거가 부족하다

예:

- 특정 회사 규정 공식이 필요한 계산 질문
- 개인 데이터가 반드시 필요한데 현재 수집/조회 경로가 없는 질문
- 문서 근거 없이 제도 계산을 해야 하는 질문

즉 `unsupported`는 실패가 아니라, `지금 구조에서 무리하게 답하지 않기 위한 안전장치`다.

## 8. 1차 generic workflow 후보

이번 Phase 6에서 바로 고려할 수 있는 범용 workflow는 아래와 같다.

### 8-1. `date_calculation`

용도:

- 날짜 차이 계산
- 기간 계산
- 시작일/종료일 기반 산정

예:

- `두 날짜 사이 며칠이야?`
- `작년 말부터 오늘까지 몇 개월이야?`

활용 core:

- `parse_date`
- `days_between`
- `months_between`
- `years_between`
- `ensure_date_order`

### 8-2. `amount_calculation`

용도:

- 합계
- 평균
- 차이
- 단순 수치 산정

예:

- `이 세 금액 평균이 얼마야?`
- `총합이 얼마야?`

활용 core:

- `parse_int_like`
- `parse_money`
- `sum_amounts`
- `average_amount`

### 8-3. `table_lookup`

용도:

- 표/엑셀/테이블 문서에서 값 찾기
- 특정 행/열/헤더 기반 조회

예:

- `표에서 A항목 금액 찾아줘`
- `3행 2열 값을 알려줘`

필요 방향:

- 테이블 추출 결과를 읽는 helper
- row/column 조건 해석
- 미매칭 시 missing/unsupported 판단

### 8-4. `document_compare`

용도:

- 두 문서의 특정 항목 차이 비교
- 조항/수치 비교

예:

- `이 두 문서의 금액 기준이 어떻게 달라?`
- `A 문서와 B 문서 차이만 정리해줘`

필요 방향:

- 두 source 확보
- 비교 기준 항목 정리
- 결과 구조화

### 8-5. `conditional_reasoning`

용도:

- 조건 분기형 정형 질문

예:

- `A면 B고 아니면 C야?`
- `이 조건이면 어느 쪽 규칙을 적용해?`

필요 방향:

- 조건식 파악
- rule branch 평가
- 결과 설명

### 8-6. `multi_source_summary`

용도:

- 여러 자료를 읽고 정해진 형식으로 요약

예:

- `이 세 문서에서 공통점만 뽑아줘`
- `자료들을 종합해서 핵심만 정리해줘`

이건 비교적 workflow와 agent의 경계에 가까우므로, 범위를 좁혀서 정형 요약형으로만 시작하는 편이 좋다.

## 9. 추천 구현 구조

권장 패키지 구조는 아래와 같다.

```text
chat/
  workflows/
    core/
      ...
    domains/
      __init__.py
      general/
        __init__.py
        date_calculation.py
        amount_calculation.py
        table_lookup.py
        document_compare.py
        conditional_reasoning.py
        multi_source_summary.py
      registry.py
      dispatch.py
```

설명:

- `general/`
  - 범용 workflow domain 구현
- `registry.py`
  - `workflow_key` 목록과 메타정보
- `dispatch.py`
  - `workflow_key -> workflow implementation` 연결

이 구조를 쓰면 나중에 원할 때만 아래를 추가할 수 있다.

```text
chat/workflows/domains/company/
chat/workflows/domains/hr/
```

즉 지금은 generic으로 가고, 나중에만 company domain을 별도 추가할 수 있다.

## 10. Phase 6 Step 0에서 먼저 정리할 것

Step 0의 목적은 구현 전에 기준을 확정하는 것이다.

### 10-1. 회사 전용 예시/키워드 정리

현재 일부 문서/주석/라우팅 fallback에는 회사 전용 예시가 남아 있다.

예:

- `퇴직금`
- `연차 계산`
- `급여`
- `수당`

Phase 6에서는 이것을 generic 기준으로 정리한다.

예시 방향:

- `날짜 차이`
- `합계`
- `평균`
- `표 조회`
- `비교`
- `조건 분기`

### 10-2. `workflow_key` 목록 확정

최소한 아래는 Step 0에서 확정해야 한다.

- `date_calculation`
- `amount_calculation`
- `table_lookup`
- `document_compare`
- `conditional_reasoning`
- `multi_source_summary`

### 10-3. route 결정 결과 확장

라우터는 앞으로 아래 중 최소 일부를 반환할 수 있어야 한다.

- `route`
- `workflow_key`
- `route_reason`
- `matched_rules`

처음부터 코드 구현을 다 하지 않더라도, 설계상 이 구조를 확정해 두는 것이 좋다.

### 10-4. unsupported 기준 확정

아래를 문서화해야 한다.

- 어떤 질문은 agent로 넘긴다
- 어떤 질문은 unsupported로 막는다
- 어떤 질문은 추가 정보 요청으로 돌린다

이 기준이 없으면 workflow와 agent 경계가 흐려진다.

## 11. 실행 흐름

Phase 6 이후 workflow 경로의 이상적인 흐름은 아래와 같다.

```text
question
-> router
-> route=workflow, workflow_key=...
-> dispatch(workflow_key)
-> selected workflow prepare
-> validate
-> execute
-> WorkflowResult
-> response layer
```

즉 Phase 5의 core는 그대로 쓰고, Phase 6은 그 위에 실제 domain workflow를 얹는 구조다.

## 12. 구현 순서 제안

### Step 0. 기준 정리

- 회사 전용 예시/키워드 정리
- generic workflow 목록 확정
- `workflow_key` 체계 정의
- `agent / unsupported` 경계 정의

### Step 1. domain 골격 추가

- `chat/workflows/domains/general/` 생성
- `registry.py` 추가
- `dispatch.py` 추가

### Step 2. workflow_key 기반 연결 구조 만들기

- registry에 workflow 메타정보 등록
- dispatch에서 key에 맞는 workflow 선택

### Step 3. 1차 generic workflow 구현

권장 순서:

1. `date_calculation`
2. `amount_calculation`
3. `table_lookup`

이 세 개가 가장 명확하고 core 재사용성이 높다.

### Step 4. route와 workflow 연결

- `workflow` route가 실제 domain workflow를 타게 연결
- 아직 지원하지 않는 key는 `unsupported` 또는 fallback으로 처리

### Step 5. 테스트

- workflow별 happy path
- missing input
- invalid input
- unsupported
- route -> workflow dispatch

---

## 13. 완료 기준

Phase 6이 완료되었다고 보려면 최소한 아래가 충족돼야 한다.

- generic workflow domain 최소 2~3개가 실제 동작한다
- `workflow_key` 체계가 정리되어 있다
- `workflow` route가 더 이상 무조건 `single_shot`으로만 포워딩되지 않는다
- `agent`와 `unsupported` 기준이 문서와 코드에 반영된다
- 회사 전용 계산식 없이도 범용 multi-step 질문 일부를 안정적으로 처리할 수 있다

---

## 14. 한 줄 정리

Phase 6은 `회사 업무 workflow를 붙이는 단계`가 아니라, `Phase 5의 범용 core 위에 질문 유형별 generic workflow domain을 올리는 단계`다.
