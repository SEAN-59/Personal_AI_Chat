# 0.4.0 Phase 3 기존 코드 개편 개발 설계

## 1. 목적

Phase 3의 목적은 Phase 2에서 LangGraph 진입점을 확보한 이후, 현재 서비스 코드를 `LangGraph 친화적 구조`로 점진 재배열하고, 프레임워크가 더 잘 담당할 orchestration 영역과 서비스 고유 로직을 분리하는 것이다.

이 단계의 핵심 목표는 아래 4가지다.

1. 현재 single-shot 파이프라인에서 orchestration과 도메인 로직의 경계를 분리한다.
2. 이후 `single-shot / workflow / agent` 3분류 확장을 고려한 패키지 구조를 만든다.
3. Phase 2에서 임시 래핑한 기존 코드를 graph-native 구조로 점진 치환할 준비를 한다.
4. 프레임워크가 대체 가능한 직접 구현 코드와 끝까지 유지할 서비스 고유 코드를 식별한다.

---

## 2. 현재 상태

Phase 2가 끝난 시점의 가정 상태는 아래와 같다.

- view는 `graph.invoke(...)`를 통해 질문 실행을 시작한다.
- graph 내부에는 최소한 `router -> single_shot` 구조가 존재한다.
- `single_shot node`는 내부에서 기존 `answer_question()`을 호출한다.
- 즉, 실행 진입점만 LangGraph로 옮겨졌고, 실제 single-shot 구현 대부분은 여전히 `query_pipeline.py` 중심으로 남아 있다.

이 상태의 특징은 다음과 같다.

- LangGraph는 들어왔지만 내부 책임 분리는 아직 약하다.
- `query_pipeline.py`가 여전히 많은 역할을 동시에 가진다.
- 이후 workflow와 agent를 붙이려면 현재 service 책임을 재정렬해야 한다.

따라서 Phase 3은 기능 확장보다 먼저, `기존 코드를 future-proof한 구조로 다시 정리하는 단계`다.

---

## 3. 이번 Phase의 범위

이번 Phase에서는 아래까지만 한다.

- 현재 single-shot 관련 service 책임 분석
- orchestration/도메인 로직 분리 기준 정의
- graph 친화적 디렉토리 구조 설계
- 기존 `query_pipeline.py`의 역할 분해 계획 수립
- graph node에서 재사용할 단위 함수 정리
- 중복 orchestration 정리 계획 수립

이번 Phase에서 하지 않는 것:

- 실제 workflow 도메인 로직 구현
- 실제 agent 구현
- 정교한 라우팅 정책 완성
- 문서 처리 파이프라인 교체
- LangChain 도입

즉, 이 단계는 `기존 코드를 graph 위에서 오래 버티는 구조로 재배열하는 단계`다.

---

## 4. 왜 이 단계가 필요한가

Phase 2까지만 끝난 상태는 구조적으로 다음 한계를 가진다.

- graph가 있어도 내부는 여전히 큰 함수 하나에 의존할 수 있다.
- 이후 workflow/agent 추가 시 기존 single-shot과 충돌 가능성이 높다.
- 프레임워크가 담당할 실행 흐름과 서비스 고유 로직이 섞여 있으면 LangGraph 도입 효과가 반감된다.

즉, Phase 3은 `LangGraph를 붙였으니 끝`이 아니라, `LangGraph를 제대로 쓸 수 있는 코드 구조`를 만드는 단계라고 봐야 한다.

---

## 5. 개편 원칙

### 5-1. orchestration은 얇게, 도메인 로직은 두껍게

이 단계의 가장 중요한 원칙이다.

LangGraph가 맡아야 할 영역:

- state 흐름
- node 연결
- route 전이
- graph 진입/종료
- step 분기

서비스 고유 코드로 남겨야 할 영역:

- 검색 함수
- 재정렬 함수
- 프롬프트 조립 함수
- 계산 함수
- 도메인 규칙
- 라우팅 기준

즉, graph는 흐름을 관리하고 실제 비즈니스 판단은 작은 함수들로 남겨야 한다.

### 5-2. 기존 코드를 한 번에 재작성하지 않는다

Phase 3은 대규모 재작성보다 `점진 치환`에 가깝다.

권장 흐름:

1. 현재 큰 함수 내부 역할 식별
2. 독립 가능한 부분 추출
3. 추출한 부분을 service/helper 함수로 이동
4. node는 조합자 역할만 하도록 얇게 정리

즉, `처음엔 래핑 -> 점진 분해 -> 최종 graph-native 조합` 순서를 유지한다.

### 5-3. single-shot도 graph 구조 기준으로 다시 본다

지금 single-shot은 기능상 단순하지만, 구조적으로는 아래 단계를 가진다.

- retrieval
- rerank
- QA cache/hit
- prompt build
- model call
- postprocess
- log/save

Phase 3에서는 이 단계를 명시적으로 바라보고, 이후 node 또는 helper 단위로 분해 가능한 형태로 정리한다.

### 5-4. workflow와 agent를 위한 자리만 만든다

이 Phase에서는 workflow나 agent를 실제 구현하지 않는다.

대신 아래를 준비한다.

- `workflows/` 디렉토리 자리
- graph/router 확장 포인트
- shared util 위치
- 향후 node/route 추가가 가능한 구조

즉, `빈 방을 만들어두는 단계`라고 보면 된다.

---

## 6. 현재 코드에서 재배치가 필요한 영역

### 6-1. `chat/services/query_pipeline.py`

현재 이 파일은 아래 역할을 동시에 가진다.

- 검색 orchestration
- QA cache 판단
- prompt build 호출
- OpenAI 호출
- token usage 저장
- chat log 저장
- source 정리

즉 역할이 많다. Phase 3에서는 이 파일을 아래 중 하나로 바꾸는 방향을 본다.

1. `single-shot 조합기` 역할만 남김
2. 내부 책임을 helper 함수로 분리
3. 이후 graph node에서 필요한 단위만 선택 사용 가능하게 만듦

### 6-2. `chat/views/message.py`

Phase 2 이후에는 graph invoke로 진입점만 바뀌지만, 여전히 아래 경계가 중요하다.

- 요청 파싱
- history 로드/저장
- graph 호출
- 에러를 HTTP 응답으로 매핑

Phase 3에서는 view가 orchestration 로직을 다시 먹지 않도록 경계를 더 분명히 해야 한다.

### 6-3. `chat/services/prompt_builder.py`

Prompt 분리가 끝난 뒤에는 single-shot 조립기 역할이 더 명확해진다.

Phase 3에서는 이 파일이

- pure helper인지
- single-shot 전용 builder인지
- 이후 workflow/agent와 공유 가능한 builder인지

를 명확히 정리해야 한다.

### 6-4. history 관련 처리

현재 `history_service.py`는 세션 히스토리 저장을 담당한다.

Phase 3에서는 아래를 정리한다.

- history는 view에서 저장할지
- graph 실행 결과에 따라 저장 여부를 누가 판단할지
- 실패 시 history 미저장 원칙 유지 여부

즉, LangGraph 도입 후에도 history 책임 경계가 흔들리지 않게 해야 한다.

## 7. 목표 패키지 구조

Phase 3 이후 목표 구조는 아래 방향을 권장한다.

```text
chat/
  graph/
    app.py
    state.py
    nodes/
      router.py
      single_shot.py
  services/
    history_service.py
    prompt_loader.py
    prompt_registry.py
    single_shot/
      __init__.py
      pipeline.py
      retrieval.py
      qa_cache.py
      prompting.py
      llm.py
      postprocess.py
  workflows/
    __init__.py
    core/
      __init__.py
    domains/
      __init__.py
```

설명:

- `graph/`
  - 실행 흐름 전용

- `services/single_shot/`
  - 현재 single-shot 세부 책임을 나누는 위치

- `workflows/core/`
  - 향후 범용 workflow 유틸 자리

- `workflows/domains/`
  - 향후 domain workflow 자리

현재 프로젝트 스타일을 고려하면 `services/workflows/`로 넣는 대안도 가능하지만, 장기적으로는 `chat/workflows/` 분리가 더 읽기 쉽다.

## 8. Single-shot 분해 방향

### 8-1. 분해 대상 단계

현재 single-shot을 아래 단계로 명시한다.

1. `retrieve_documents(question)`
2. `rerank_documents(question, candidates)`
3. `find_canonical_qa(question)`
4. `resolve_qa_cache(qa_hits)`
5. `build_single_shot_messages(...)`
6. `run_single_shot_llm(messages)`
7. `save_single_shot_side_effects(...)`
8. `build_single_shot_result(...)`

이 단계가 실제로 곧바로 파일 하나씩으로 분리될 필요는 없다.

하지만 최소한 `query_pipeline.py` 안에서라도 역할을 함수 단위로 끊어야 이후 node 단위 조합이 쉬워진다.

### 8-2. 왜 이렇게 쪼개는가

이후를 생각하면 아래 이점이 있다.

- workflow에서도 retrieval만 재사용 가능
- agent에서도 llm 호출 래퍼만 재사용 가능
- QA cache 전략을 따로 교체하기 쉬움
- side effect 저장 규칙을 테스트하기 쉬움

즉, 단순 코드 정리가 아니라 재사용 구조 확보 목적이다.

---

## 9. Graph-native 치환 방향

Phase 3에서는 아직 full graph-native로 가지 않는다. 다만 치환 방향을 명시한다.

현재:

- `single_shot node -> answer_question()`

중간 목표:

- `single_shot node -> single_shot.pipeline.run(...)`

그 다음 목표:

- `single_shot node -> retrieval/prompt/llm/postprocess helper 조합`

즉 Phase 3은 두 번째 단계에 해당한다.

핵심은 node가 거대한 black box를 호출하는 수준에서 벗어나, 내부 책임을 더 잘 보이게 만드는 것이다.

---

## 10. 공용 규칙 정리 대상

Phase 3에서는 아래 공용 규칙도 함께 정리해야 한다.

- 에러 매핑 규칙
- history 저장 규칙
- side effect 저장 위치
- token usage 기록 규칙
- source 표시 규칙

이유:

single-shot / workflow / agent가 모두 생기면 이런 공통 규칙이 각 경로마다 중복되기 쉽기 때문이다.

이 단계에서 완성까지는 아니더라도, 최소한 `공통 규칙 목록`은 문서화하고 위치를 정해야 한다.

## 12. 세부 개발 순서

### Step 1. 현재 single-shot 책임 맵 작성

할 일:

- `query_pipeline.py` 내부 역할을 단계별로 분해해 목록화
- 어떤 부분이 orchestration이고 어떤 부분이 business logic인지 분리

완료 기준:

- 현재 함수가 무엇을 얼마나 맡고 있는지 설명 가능한 상태

### Step 2. single-shot 내부 helper 분리

할 일:

- 검색 관련 helper 추출
- QA cache 관련 helper 추출
- prompt build 호출 경계 정리
- LLM 호출 래퍼 경계 정리

완료 기준:

- single-shot pipeline이 더 작은 함수 조합으로 읽히기 시작함

### Step 3. single-shot 패키지 구조 도입

할 일:

- `services/single_shot/` 구조 생성
- 분리한 helper를 적절한 위치로 이동

완료 기준:

- `query_pipeline.py`의 책임이 줄고, single-shot 관련 코드가 의미 있는 단위로 나뉨

### Step 4. graph node와 single-shot 서비스 경계 재정리

할 일:

- node는 state 입출력 중심
- service는 실제 처리 중심
- 두 레이어의 책임 구분 문서화

완료 기준:

- `single_shot node`가 지나치게 많은 로직을 갖지 않음

### Step 5. 공통 규칙 목록 정리

할 일:

- history 저장 규칙
- error 처리 규칙
- token usage 기록 규칙
- source 처리 규칙

완료 기준:

- 이후 workflow/agent에서도 재사용 가능한 공통 규칙 목록이 생김

### Step 6. workflows 디렉토리 골격 생성

할 일:

- `chat/workflows/core`
- `chat/workflows/domains`

골격 생성

완료 기준:

- Phase 5, 6 작업을 받을 구조가 미리 준비됨

### Step 7. 회귀 점검

할 일:

- 기존 single-shot 질문셋 재실행
- JSON 응답 유지 확인
- history 저장 동작 확인
- source 노출 및 chat_log_id 유지 확인

완료 기준:

- 구조는 바뀌었지만 사용자 관점 동작은 그대로 유지됨

## 13. 예상 위험 요소

### 13-1. 너무 많이 분해해서 오히려 복잡해질 수 있음

위험:

- 파일만 많아지고 책임은 더 안 보일 수 있다.

대응:

- 역할이 분명한 경우에만 분리
- 의미 없는 1함수 1파일 분리는 피함

### 13-2. helper 분리 중 side effect 위치가 섞일 수 있음

위험:

- ChatLog 저장, TokenUsage 저장, source 생성 시점이 어긋날 수 있다.

대응:

- side effect는 한 계층에 모아 관리
- 저장 규칙을 먼저 문서화

### 13-3. graph와 service 경계가 다시 무너질 수 있음

위험:

- node에서 business logic를 다시 많이 작성하게 될 수 있다.

대응:

- node는 state 입출력과 route 흐름만 담당
- 실제 처리 코드는 service/helper로 유지

### 13-4. workflow/agent 자리만 만들고 실제 재사용성이 없을 수 있음

위험:

- 디렉토리만 늘고 실제 공유 규칙이 정리되지 않을 수 있다.

대응:

- 공통 규칙과 helper 재사용 포인트를 함께 정리

---

## 14. 완료 정의

Phase 3은 아래 조건을 만족하면 완료로 본다.

- single-shot 관련 코드가 더 작은 책임 단위로 정리되었다.
- graph node와 service/helper의 경계가 분명해졌다.
- `query_pipeline.py`의 책임이 줄었다.
- 향후 workflow/agent를 넣을 디렉토리와 구조가 준비되었다.
- 현재 사용자 관점 동작은 유지된다.

---

## 15. Phase 3 완료 후 다음 단계 연결

Phase 3이 끝나면 다음 단계인 `질문 라우팅 설계`와 `Workflow Core 개발`을 더 자연스럽게 진행할 수 있다.

- router가 어떤 service를 호출할지 명확해짐
- workflow core가 들어갈 위치가 준비됨
- single-shot과 workflow의 공통/차별 지점을 정의하기 쉬워짐

즉 Phase 3은 `기존 코드를 정리하는 작업`이면서 동시에, `0.4.0 구조를 오래 유지할 수 있게 만드는 핵심 기반 작업`으로 본다.
