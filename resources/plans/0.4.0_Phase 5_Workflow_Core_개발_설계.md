# 0.4.0 Phase 5 Workflow Core 개발 설계

## 1. 목적

Phase 5의 목적은 `workflow` 경로에서 공통으로 재사용할 수 있는 계산, 정규화, 검증, 포맷팅 기능을 `Workflow Core`로 분리하는 것이다.

이 단계의 핵심 목표는 아래 4가지다.

1. 특정 회사 규칙과 무관한 공통 workflow 기능을 정의한다.
2. 이후 여러 workflow가 공유할 `core` 계층을 만든다.
3. workflow와 agent가 공통으로 활용 가능한 계산/정규화 기반을 준비한다.
4. Phase 6의 도메인 workflow 구현 전에 `재사용 가능한 범용 처리 레이어`를 확립한다.

---

## 2. 현재 상태

Phase 4가 끝난 시점의 가정 상태는 아래와 같다.

- 질문은 `single_shot / workflow / agent` 중 하나로 라우팅될 수 있다.
- workflow route는 정의되었지만, 실제 workflow 본체는 아직 구현되지 않았다.
- Phase 3을 통해 `chat/workflows/`를 받을 구조는 어느 정도 정리되어 있다.

이 시점에서 바로 `퇴직금`, `연차`, `수당` 같은 도메인 workflow를 만들면, 공통 계산과 도메인 규칙이 뒤섞일 가능성이 높다.

따라서 Phase 5에서는 먼저 아래를 공통 레이어로 빼낸다.

- 날짜 처리
- 기간 계산
- 수치 계산
- 값 정규화
- 입력 검증
- workflow 실행 결과 포맷팅

즉, 이 단계는 `업무 로직을 만들기 전, 공통 엔진을 먼저 만드는 단계`다.

---

## 3. 이번 Phase의 범위

이번 Phase에서는 아래를 한다.

- workflow core 책임 정의
- 공통 유틸 함수 목록 정의
- 입력/출력 데이터 구조 정의
- 공통 validation 규칙 정의
- 공통 결과 포맷 구조 정의
- `chat/workflows/core/` 패키지 설계

이번 Phase에서 하지 않는 것:

- 퇴직금 공식 구현
- 연차 산정 규칙 구현
- 회사 전용 정책 계산
- agent용 ReAct loop 구현
- 외부 인사/급여 시스템 연동

즉, 이 단계는 `workflow 공용 부품 설계 및 최소 구현 준비 단계`다.

---

## 4. Workflow Core란 무엇인가

Workflow Core는 특정 도메인 규칙을 알지 못하는, 범용 처리 계층이다.

예를 들어 아래는 core에 들어갈 수 있다.

- 날짜 문자열 파싱
- 날짜 차이 계산
- 개월 수/일 수 계산
- 합계/평균 계산
- 누락값 검사
- 수치 문자열 정규화
- 결과 포맷팅

반대로 아래는 core에 들어가면 안 된다.

- 퇴직금 공식
- 연차 산정 정책
- 회사별 수당 규칙
- 조직별 예외 조항

즉 기준은 단순하다.

- 여러 workflow에서 재사용 가능하면 `core`
- 특정 업무 규칙에 묶이면 `domain`

---

## 5. 설계 원칙

### 5-1. core는 특정 도메인을 몰라야 한다

Workflow Core는 `퇴직금`, `연차`, `급여`라는 단어를 몰라도 동작해야 한다.

예:

- `calculate_day_difference(start, end)` -> core
- `calculate_retirement_pay(...)` -> domain

이 원칙이 무너지면 core가 금방 회사 전용 코드로 변질된다.

### 5-2. 순수 함수 중심으로 설계한다

core는 가능하면 아래 특성을 가져야 한다.

- 입력이 같으면 출력이 같다
- DB 접근이 없다
- 외부 API 호출이 없다
- side effect가 없다

이유:

- 테스트가 쉽다
- workflow/agent 어디서든 재사용 가능하다
- 디버깅이 쉽다

### 5-3. 문자열 입력보다 정규화된 값 중심으로 다룬다

workflow에서는 입력값이 종종 지저분하다.

예:

- `"2025.01.31"`
- `"2025-01-31"`
- `"1,234,567원"`
- `" 3 개월 "`

따라서 core의 역할은 `계산`뿐 아니라 `정규화`도 포함해야 한다.

### 5-4. 에러 대신 명시적 검증 결과를 우선 고려한다

단순 예외만 던지면 workflow가 사용자에게 어떤 값이 부족한지 설명하기 어렵다.

따라서 아래 두 패턴 중 하나를 고려한다.

1. `raise` 기반
2. `ValidationResult` 반환 기반

Phase 5에서는 최소한 아래는 문서화해야 한다.

- 어떤 상황에서 예외를 쓰는지
- 어떤 상황에서 검증 결과 객체를 쓰는지

권장:

- 프로그래밍 오류 -> 예외
- 사용자 입력 부족/형식 오류 -> 검증 결과

### 5-5. 결과 구조도 공통화한다

workflow마다 결과 형식이 제각각이면 BO나 응답 조립 시 불편하다.

따라서 core에는 최소한 다음 개념이 있어야 한다.

- 계산 결과 값
- 중간 계산 근거
- 누락 입력값
- 경고 메시지

즉 단순 숫자 하나를 반환하는 것보다, `설명 가능한 결과 구조`를 지향한다.

## 6. 목표 패키지 구조

권장 구조는 아래와 같다.

```text
chat/
  workflows/
    __init__.py
    core/
      __init__.py
      dates.py
      numbers.py
      validation.py
      result.py
      formatting.py
      base.py
    domains/
      __init__.py
```

설명:

- `dates.py`
  - 날짜 파싱/비교/차이 계산

- `numbers.py`
  - 금액/숫자 정규화, 합계, 평균

- `validation.py`
  - 필수값/형식 검증

- `result.py`
  - 공통 결과 구조

- `formatting.py`
  - 사용자 응답용 문자열 포맷 helper

- `base.py`
  - workflow 공통 타입, base interface, 실행 컨텍스트 후보

현재 Phase에서는 domains는 비워둬도 된다.

## 7. Core 기능 분류

### 7-1. 날짜 처리

목표:

- 문자열을 날짜로 파싱
- 날짜 간 차이 계산
- 개월/일/년 단위 계산
- 시작일/종료일 유효성 검증

예상 함수:

- `parse_date(value: str) -> date`
- `days_between(start, end) -> int`
- `months_between(start, end) -> int`
- `years_between(start, end) -> int`
- `ensure_date_order(start, end) -> ValidationResult`

사용될 수 있는 workflow:

- 근속기간 계산
- 휴가 산정
- 계약 기간 계산

### 7-2. 숫자 처리

목표:

- 문자열 숫자 정규화
- 금액 합계/평균 계산
- 반올림/절삭 규칙 helper

예상 함수:

- `parse_int_like(value: str) -> int`
- `parse_money(value: str) -> int`
- `sum_amounts(values: list[int]) -> int`
- `average_amount(values: list[int]) -> int | float`

사용될 수 있는 workflow:

- 평균 급여 계산
- 잔액 계산
- 정산 계산

### 7-3. 검증 처리

목표:

- 필수 입력값 존재 여부 확인
- 형식 오류 확인
- 누락 필드 목록 반환

예상 함수:

- `require_fields(data, fields) -> ValidationResult`
- `require_non_empty(value, field_name) -> ValidationResult`
- `combine_validations(results) -> ValidationResult`

사용될 수 있는 workflow:

- 퇴직금 계산 전 필수값 확인
- 연차 계산 전 입사일 존재 확인

### 7-4. 결과 구조화

목표:

- 계산 결과를 공통 객체로 반환
- 경고/누락/중간값 포함

예상 구조:

- `WorkflowResult`
  - `status`
  - `value`
  - `details`
  - `missing_fields`
  - `warnings`

상태 후보:

- `ok`
- `missing_input`
- `invalid_input`
- `unsupported`

### 7-5. 포맷팅 처리

목표:

- 숫자/날짜를 사용자 친화적으로 표시
- 결과 설명 문자열을 만들 때 필요한 helper 제공

예상 함수:

- `format_currency(value) -> str`
- `format_date(value) -> str`
- `format_duration(days=None, months=None, years=None) -> str`

주의:

- full 자연어 답변 생성은 core 책임이 아니다.
- core는 formatting helper까지만 담당하고, 최종 문장 조립은 이후 workflow/domain 또는 response layer가 맡는다.

## 8. 공통 타입 설계

### 8-1. ValidationResult

권장 개념:

```python
ValidationResult(
    ok: bool,
    missing_fields: list[str],
    errors: list[str],
)
```

용도:

- 어떤 입력이 빠졌는지 명시
- workflow가 사용자에게 추가 정보를 요청할 근거 제공

### 8-2. WorkflowResult

권장 개념:

```python
WorkflowResult(
    status: str,
    value: Any | None,
    details: dict,
    missing_fields: list[str],
    warnings: list[str],
)
```

용도:

- domain workflow가 계산 결과를 일관되게 반환
- response layer가 이를 바탕으로 답변 조립

### 8-3. WorkflowInput

모든 workflow가 같은 입력 구조를 가질 필요는 없다.

하지만 core 관점에서는 아래 감각이 필요하다.

- raw input
- normalized input
- validation result

즉, 직접 문자열로 계산하지 말고 `정규화 -> 검증 -> 계산` 순서를 갖게 해야 한다.

## 9. Base Workflow 개념

Phase 5에서 실제 abstract class까지 강제할 필요는 없다.

하지만 향후 Phase 6을 위해 아래 정도는 정의해둘 가치가 있다.

- workflow가 받는 입력
- workflow가 반환하는 결과
- validation 단계
- execution 단계

예상 개념:

```text
prepare_input
-> validate
-> execute
-> return WorkflowResult
```

즉 domain workflow도 결국 공통적으로 아래 구조를 따르게 된다.

1. 입력 정규화
2. 입력 검증
3. 계산 실행
4. 결과 구조화

## 11. 세부 개발 순서

### Step 1. core 책임 목록 확정

할 일:

- 날짜/숫자/검증/결과/포맷팅 중 무엇을 core로 둘지 확정

완료 기준:

- domain과 core 경계가 문서로 설명 가능

### Step 2. 공통 타입 정의

할 일:

- `ValidationResult`
- `WorkflowResult`

구조 정의

완료 기준:

- domain workflow가 어떤 형태로 결과를 반환할지 일관된 기준이 생김

### Step 3. 날짜 유틸 설계

할 일:

- 날짜 파싱 규칙
- 차이 계산 규칙
- 순서 검증 규칙

완료 기준:

- 기간 계산이 필요한 workflow에 공용 함수 재사용 가능

### Step 4. 숫자 유틸 설계

할 일:

- 금액 문자열 정규화
- 합계/평균 함수 정의

완료 기준:

- 계산 workflow의 공통 수치 처리 기반 확보

### Step 5. 검증 유틸 설계

할 일:

- 필수값 체크
- 형식 오류 체크
- 검증 결과 합성 방식 정의

완료 기준:

- workflow가 누락값/오류를 일관되게 표현 가능

### Step 6. formatting helper 설계

할 일:

- 금액/날짜/기간 표시용 helper 정의

완료 기준:

- workflow 응답 조립 시 공통 포맷 사용 가능

### Step 7. Base Workflow 개념 정리

할 일:

- prepare/validate/execute/result 흐름 정의
- Phase 6 domain workflow가 따를 최소 패턴 확정

완료 기준:

- domain workflow 구현 방식이 흔들리지 않음

## 12. 예상 위험 요소

### 12-1. core가 도메인 규칙을 먹어버릴 수 있음

위험:

- 공통 함수에 퇴직금/연차 정책 같은 규칙이 섞이면 core가 오염된다.

대응:

- 공통 함수는 범용 처리만 담당
- 도메인 규칙은 반드시 domains로 분리

### 12-2. 너무 많은 유틸 파일이 생길 수 있음

위험:

- 구조는 좋아 보이지만 실제로는 지나치게 쪼개질 수 있다.

대응:

- 역할이 분명한 경우에만 파일 분리
- 의미 없는 파일 증식은 피함

### 12-3. 결과 구조가 너무 단순하면 나중에 설명이 약해질 수 있음

위험:

- 단일 숫자만 반환하면 사용자가 왜 그 값이 나왔는지 설명하기 어렵다.

대응:

- `details`, `missing_fields`, `warnings`를 초기에 포함

### 12-4. 입력 검증 정책이 workflow마다 달라질 수 있음

위험:

- 어떤 workflow는 예외를 던지고 어떤 workflow는 문자열을 반환하는 식으로 일관성이 깨질 수 있다.

대응:

- core 단계에서 검증 결과 표현 방식을 먼저 통일

---

## 13. 완료 정의

Phase 5는 아래 조건을 만족하면 완료로 본다.

- `chat/workflows/core/` 구조가 정의되었다.
- 날짜/숫자/검증/결과/포맷팅의 core 책임이 구분되었다.
- `ValidationResult`, `WorkflowResult` 같은 공통 타입이 정의되었다.
- domain workflow가 따라야 할 최소 실행 패턴이 정리되었다.
- Phase 6에서 실제 업무 workflow를 올릴 수 있는 범용 기반이 준비되었다.

---

## 14. Phase 5 완료 후 다음 단계 연결

Phase 5가 끝나면 Phase 6에서 아래를 더 쉽게 진행할 수 있다.

- 퇴직금 workflow 구현
- 근속기간 workflow 구현
- 연차 계산 workflow 구현
- 공통 계산 함수 재사용
- 입력 부족/오류 응답의 일관성 유지

즉 Phase 5는 `workflow를 직접 만드는 단계`가 아니라, `workflow를 여러 개 만들어도 버틸 공통 엔진을 준비하는 단계`로 본다.
